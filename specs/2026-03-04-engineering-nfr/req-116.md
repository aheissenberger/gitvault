# REQ-116: `gitvault get` / `gitvault set` — Single Key-Value Access

## Context

Developers often need to read or update a single secret value without opening an editor or
writing a script that unseals a whole file. Common scripting needs:

```bash
# Read DB password from sealed config into an env var
export DB_PASS=$(gitvault get config/secrets.json db.password)

# Rotate a token
gitvault set config/secrets.json api.token --stdin < new-token.txt

# Read from an encrypted store file
gitvault get .gitvault/store/prod/config.json.age db.password --env prod
```

## Goal

Two focused commands for programmatic key-value access on both sealed (in-place field
encryption) and encrypted store files (`.age` archives):

- **`gitvault get <FILE> <KEY>`** — print the plaintext value of a single key to stdout
- **`gitvault set <FILE> <KEY> [VALUE]`** — update (or create) a single key's value and
  re-seal/re-encrypt atomically

## Non-goals

- Bulk/multi-key operations (use `seal`/`unseal` or `edit` for those).
- Creating deeply nested key paths (only top-level upsert for new keys).
- Format conversion (the file format is determined by extension and preserved).

## Constraints

- TOML files MUST use `toml_edit` for structure-preserving updates (no round-trip loss).
- Plaintext values MUST be zeroized before return.
- Value must never appear in process args when using `--stdin` (read from stdin, not args).
- No new Cargo dependencies.

## Acceptance Criteria

See frontmatter `acceptance` list (AC1–AC12).

## Key Format Reference

| File type | Key example | Notes |
|-----------|-------------|-------|
| `.json` | `db.password` | Dot-path; nested objects |
| `.yaml` / `.yml` | `server.tls.cert` | Dot-path; nested mappings |
| `.toml` | `database.password` | Dot-path; nested tables |
| `.env` | `API_KEY` | Variable name only; no dots |
| `.age` (store) | Same as stem format | Detect from `filename.json.age` → JSON rules |

## Test Plan

```bash
# Seal a JSON file, get a field
gitvault seal config.json
SECRET=$(gitvault get config.json db.password)

# Update a field
echo "new-token-value" | gitvault set config.json api.token --stdin
gitvault get config.json api.token   # should print "new-token-value"

# JSON output
gitvault get config.json db.password --json
# → {"file":"config.json","key":"db.password","value":"s3cr3t"}

# Store file
gitvault get .gitvault/store/prod/config.json.age db.password --env prod
```

## Notes

- `get` is intentionally simple: no label, no quoting, raw value + newline. This mirrors
  the behaviour of `chamber read -q` and `vault kv get -field=key`.
- `set --stdin` trims exactly one trailing newline (as produced by `echo`) to match
  expected shell behaviour. Additional whitespace is preserved.
- For `.env` files, `set` must not reorder variables or remove blank lines/comments.
