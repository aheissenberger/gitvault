# REQ-118: Unified Rule Engine for Seal, Materialize, Run, and Status

## Context

Multiple command surfaces currently evaluate path and key matching with partially duplicated logic and command-local schema assumptions. This increases drift risk between `seal`, `materialize`, `run`, and `status`, and makes policy expression harder for users.

Prior schema also relied on seal-specific keys that do not generalize well to run/materialize behavior.

## Update Note (2026-03-16)

REQ-119 extends runtime source selection for `[[run.rule]]` and
`[[materialize.rule]]` with an additive optional `source` discriminator (`store` or
`sealed`) while preserving REQ-118's unified
matcher model, deterministic rule evaluation, and strict parse validation.

## Decision

Introduce one structured TOML rule model per command namespace:

```toml
[[seal.rule]]
action = "allow"
path = "config/**/*.json"
keys = ["db.*", "api.*"]

[[materialize.rule]]
action = "deny"
path = "**/*.dev.*"
dir_prefix = true
path_prefix = true
custom_prefix = "MATERIALIZE"

[[run.rule]]
action = "allow"
path = "services/**"
keys = ["*_PASSWORD", "*_TOKEN"]
dir_prefix = false
path_prefix = true
custom_prefix = "RUNTIME"
```

Rules are glob-only and field-based (`action`, `path`, `keys`, and runtime prefix fields) with no regex support and no combined delimiter notation.

## Non-Goals

- No regex engine or regex fallback.
- No backward-compatible parser for legacy seal matcher keys.
- No mixed-mode config where legacy and rule-table schema coexist.

## Migration

Configs using legacy matcher keys must be updated to structured rules. When legacy keys are present, gitvault returns a usage/config parse error that points users to `[[seal.rule]]`, `[[materialize.rule]]`, and `[[run.rule]]`.

## Rationale

- Consistent policy expression across commands.
- Better DX through one mental model.
- Deterministic and testable matcher behavior.
- Simpler future extensions without introducing new one-off matcher sections.

## Implementation Notes

- Shared matcher behavior is implemented in `src/matcher.rs` and imported by all command modules.
- Rule evaluation is deterministic and order-sensitive.
- Paths are normalized to `/` before glob matching for cross-platform parity.
- Key matching uses the same glob semantics everywhere.

## Supersession

This requirement supersedes REQ-117 (`S-20260304-R117`) as the normative source for multi-format materialize/run matcher behavior.
