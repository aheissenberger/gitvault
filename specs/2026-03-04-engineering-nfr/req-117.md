# REQ-117: `decrypt_env_secrets` — multi-format store file injection

## Problem

`gitvault run` fails with `Invalid .env content: Error parsing line: '{'` when the
`.gitvault/store/<env>/` directory contains non-env store files (e.g. `more.json.age`,
`conf/dbsecrets.json.age`). The root cause is that `decrypt_env_secrets` unconditionally
passes every decrypted file through `merge::parse_env_pairs`, which only handles
`KEY=value` format. JSON, YAML, and TOML content produces a parse error at the first `{`,
`-`, or `[` character.

## Fix

Detect the format from the `.age` filename stem using the unified `validated_extension`
helper (introduced in REQ-116 / fix: unify env file format detection). Route each file
through the appropriate parser:

| Stem format | Parser |
|---|---|
| `.env`, `.env.<suffix>`, `<name>.env` | `merge::parse_env_pairs` (existing) |
| `*.json` | serde_json — flatten leaf scalars |
| `*.yaml` / `*.yml` | serde_yml — flatten leaf scalars |
| `*.toml` | toml — flatten leaf scalars |
| unknown | skip + stderr warning |

## Flattening rules

JSON/YAML/TOML objects are recursively flattened into `KEY=value` pairs:
- Path segments are joined with `_` and uppercased.
- Nested: `{"db": {"password": "s"}}` → `DB_PASSWORD=s`
- Scalars: strings, integers, floats, booleans → stringified.
- Arrays: skipped entirely (consistent with `seal` AC2).
- Null values: skipped.
