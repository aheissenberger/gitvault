# REQ-112: `seal` and `unseal` — In-Place Field-Level Encryption Commands

**ID:** S-20260304-R112
**Status:** Proposed
**Priority:** High
**Category:** Breaking Change / Developer Experience

## Problem

gitvault currently has two encryption models that silently diverge based on flags:

- `encrypt <file>` → writes to `.gitvault/store/<env>/<file>.age` (store model)
- `encrypt <file> --fields key` → modifies `<file>` in-place (in-place model)
- `encrypt <file> --value-only` → modifies `<file>` in-place (in-place model)

The same verb does fundamentally different things. A user who learns `encrypt` for whole files
is confused when `--fields` silently changes the output location. A user who learns
`--value-only` for `.env` cannot apply the same mental model to JSON.

**SOPS resolves this by having only one model** (always in-place). gitvault has two legitimate
models (whole-file archive for multi-env isolation; in-place for structured files). The solution
is to make each model a distinct top-level command with a distinct verb.

## Decision

Introduce `seal` and `unseal` as the canonical commands for **in-place field-level
encryption** of structured files. The existing `encrypt`/`decrypt` commands are reserved
exclusively for **whole-file archive encryption** (writing to/reading from `.gitvault/store/`).

| Verb | Model | Output location |
|---|---|---|
| `seal` | In-place, field-level | Same file path |
| `unseal` | In-place, field-level | Same file path (or stdout with `--reveal`) |
| `encrypt` | Whole-file archive | `.gitvault/store/<env>/<mirrored-path>.age` |
| `decrypt` | Whole-file archive | Original source path (resolved from store) |

## Command Specification

### `gitvault seal`

```
gitvault seal [OPTIONS] <FILE>
```

| Flag | Default | Description |
|---|---|---|
| `<FILE>` | — | JSON, YAML, TOML, or .env file to seal in-place |
| `--fields <FIELDS>` | all eligible values | Comma-separated dot-path keys to encrypt. Omit to encrypt all string values |
| `-r, --recipient <PUBKEY>` | `.gitvault/recipients/` | age public key (repeat for multi-recipient) |
| `-i, --identity <IDENTITY>` | — | Identity key file (also `GITVAULT_IDENTITY`) |
| `-e, --env <ENV>` | — | Environment (used for recipient resolution only) |

**File format dispatch:**

| Extension | Mechanism | Idempotency check |
|---|---|---|
| `.json` | age ASCII-armor per string leaf | `is_age_armor(value)` |
| `.yaml` / `.yml` | age ASCII-armor per string leaf | `is_age_armor(value)` |
| `.toml` | age ASCII-armor per string leaf | `is_age_armor(value)` |
| `.env` / `.env.*` | `age:` prefix + binary base64 | `is_env_encrypted(value)` |

### `gitvault unseal`

```
gitvault unseal [OPTIONS] <FILE>
```

| Flag | Default | Description |
|---|---|---|
| `<FILE>` | — | Sealed file to unseal |
| `--reveal` | off | **Recommended.** Print plaintext to stdout; never write to disk |
| `--fields <FIELDS>` | all encrypted values | Comma-separated dot-path keys to decrypt |
| `-i, --identity <IDENTITY>` | — | Identity key file (also `GITVAULT_IDENTITY`) |

## `.gitvault/sealed.toml` index

```toml
# .gitvault/sealed.toml — committed with the repo; managed by gitvault seal/unseal/rekey

[files]
"config/app.json"   = { fields = "all", sealed_at = "2026-03-04T20:00:00Z" }
"helm/values.yaml"  = { fields = ["postgresql.auth.password", "redis.auth.password"], sealed_at = "2026-03-04T20:01:00Z" }
".env"              = { fields = "all", sealed_at = "2026-03-04T20:02:00Z" }
```

- `"all"` means the file was sealed without `--fields` (all eligible values are encrypted).
- An explicit array means only those fields are encrypted; other values may be plaintext.
- `gitvault status` reads this index and checks that each listed file has at least one
  encrypted value in its recorded fields. Drift (unsealed plaintext) is reported.
- `gitvault rekey` re-seals every file in the index with the current recipients.

## Migration Path

```bash
# Before → After

gitvault encrypt app.json --fields db.pass --env dev
→ gitvault seal app.json --fields db.pass

gitvault encrypt .env --value-only --env dev
→ gitvault seal .env

gitvault decrypt .gitvault/store/dev/app.json.age --fields db.pass --reveal
→ gitvault unseal app.json --fields db.pass --reveal

gitvault decrypt .gitvault/store/dev/.env.age --value-only
→ gitvault unseal .env
```

## Test Plan

- Roundtrip: `seal` then `unseal --reveal` returns original content for JSON/YAML/TOML/.env.
- Idempotency: `seal` on already-sealed file produces identical ciphertext (REQ-5).
- `--fields` narrow: only listed fields change; others untouched.
- `unseal` in-place: plaintext written to file; warning emitted to stderr.
- `unseal --reveal`: stdout contains plaintext; file on disk unchanged.
- Unsupported extension: usage error with guidance.
- `sealed.toml` written on first seal; updated on subsequent seals.
- `status` detects drift when a sealed file has been unsealed in-place.
- `rekey` re-seals all files in `sealed.toml` with new recipients.
