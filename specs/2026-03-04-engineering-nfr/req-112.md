# REQ-112: `seal` and `unseal` — In-Place Field-Level Encryption Commands

**ID:** S-20260304-R112
**Status:** Proposed
**Priority:** High
**Category:** Breaking Change / Developer Experience

## Problem

gitvault currently has two encryption models that silently diverge based on flags:

- `encrypt <file>` → writes to `.gitvault/store/<env>/<file>.age` (store model)
- `encrypt <file> --fields key` → modifies `<file>` in-place (in-place model)
- `encrypt <file> --value-only` → modifies `<file>` in-place (in-place model)

The same verb does fundamentally different things. A user who learns `encrypt` for whole files
is confused when `--fields` silently changes the output location. A user who learns
`--value-only` for `.env` cannot apply the same mental model to JSON.

**SOPS resolves this by having only one model** (always in-place). gitvault has two legitimate
models (whole-file archive for multi-env isolation; in-place for structured files). The solution
is to make each model a distinct top-level command with a distinct verb.

## Decision

Introduce `seal` and `unseal` as the canonical commands for **in-place field-level
encryption** of structured files. The existing `encrypt`/`decrypt` commands are reserved
exclusively for **whole-file archive encryption** (writing to/reading from `.gitvault/store/`).

| Verb | Model | Output location |
|---|---|---|
| `seal` | In-place, field-level | Same file path |
| `unseal` | In-place, field-level | Same file path (or stdout with `--reveal`) |
| `encrypt` | Whole-file archive | `.gitvault/store/<env>/<mirrored-path>.age` |
| `decrypt` | Whole-file archive | Original source path (resolved from store) |

## Non-Goals

The following are explicitly out of scope for this specification:

- **Binary file sealing** — `seal`/`unseal` operate only on UTF-8 text formats (JSON, YAML, TOML, .env). For binary files, use `encrypt`/`decrypt`.
- **Recursive directory sealing** — `seal <DIR>` is not supported; callers must enumerate files explicitly or use `[seal].patterns` glob matching via `status`.
- **Streaming / piped input** — `seal` always reads from a named file path; stdin is not a valid source for in-place sealing.
- **Partial rollback on failure** — mid-run failure leaves the file either fully sealed (write succeeded before error) or unchanged (error occurred before write). Partial sealing — some fields sealed, others not — cannot occur because `seal` performs an in-memory transformation of all fields first and then writes the result in a single atomic file operation. The only non-atomic step is the subsequent `config.toml` update, which is handled by AC13's recovery-fragment mechanism.
- **Automatic unseal on git checkout** — triggering `unseal` as a git smudge filter is a separate concern (REQ-future); this spec does not define smudge/clean filter integration.
- **Non-age encryption backends** — `seal` always uses age encryption; no other backend is planned.
- **Windows path separators in patterns** — `[seal].patterns` always uses forward slashes (POSIX glob); Windows users must use forward slashes in their `config.toml`.

## Command Specification

### `gitvault seal`

```
gitvault seal [OPTIONS] <FILE>
```

| Flag | Default | Description |
|---|---|---|
| `<FILE>` | — | JSON, YAML, TOML, or .env file to seal in-place |
| `--fields <FIELDS>` | all eligible values | Comma-separated dot-path keys to encrypt. Omit to encrypt all string values |
| `-r, --recipient <PUBKEY>` | `.gitvault/recipients/` | age public key (repeat for multi-recipient) |
| `-i, --identity <IDENTITY>` | — | Identity key file (also `GITVAULT_IDENTITY`) |
| `-e, --env <ENV>` | — | Environment (used for recipient resolution only) |

**File format dispatch:**

| Extension | Mechanism | Idempotency check |
|---|---|---|
| `.json` | age ASCII-armor per string leaf | `is_age_armor(value)` |
| `.yaml` / `.yml` | age ASCII-armor per string leaf | `is_age_armor(value)` |
| `.toml` | age ASCII-armor per string leaf | `is_age_armor(value)` |
| `.env` / `.env.*` | `age:` prefix + binary base64 | `is_env_encrypted(value)` |

### `gitvault unseal`

```
gitvault unseal [OPTIONS] <FILE>
```

| Flag | Default | Description |
|---|---|---|
| `<FILE>` | — | Sealed file to unseal |
| `--reveal` | off | **Recommended.** Print plaintext to stdout; never write to disk |
| `--fields <FIELDS>` | all encrypted values | Comma-separated dot-path keys to decrypt |
| `-i, --identity <IDENTITY>` | — | Identity key file (also `GITVAULT_IDENTITY`) |

## `.gitvault/config.toml` — seal index (Option C, pattern-based)

Rather than a separate `sealed.toml` state file, the index of sealed files is declared as
**intent patterns** in the existing `.gitvault/config.toml`. This eliminates merge conflicts:
two branches that each seal a different file produce two independent `[[seal.override]]` blocks
or two entries appended to `patterns` — both merge cleanly as they do not touch the same TOML
key.

### Schema

```toml
# .gitvault/config.toml

[seal]
# Files matching these glob patterns are expected to have all eligible string values sealed.
# gitvault seal auto-appends here when a sealed file is not already covered.
patterns = [
  "**/.env*",           # all .env variants anywhere in the repo
  "config/**/*.json",   # all JSON under config/
  "helm/values.yaml",   # exact path
]

# Per-pattern field overrides: only the listed fields are expected to be sealed.
# When --fields is used on a file covered by a wildcard pattern, seal auto-appends here.
[[seal.override]]
pattern = "helm/values.yaml"
fields  = ["postgresql.auth.password", "redis.auth.password"]

# Files matching these patterns are never flagged by status even if unsealed.
[[seal.exclude]]
pattern = "config/public.json"
```

### Auto-update rules (enforced by `seal`)

| Situation | Action taken by `seal` |
|---|---|
| `config.toml` absent | Create file with minimal `[seal]\npatterns = ["<path>"]` |
| File not covered by any `patterns` entry, no `--fields` | Append `"<relative-path>"` to `patterns` |
| File not covered by any `patterns` entry, with `--fields` | Append `"<relative-path>"` to `patterns` **and** append `[[seal.override]]` block |
| File covered by a wildcard pattern, with `--fields`, no override exists | Append `[[seal.override]]` with exact path and fields |
| File covered by a wildcard pattern, with `--fields`, override exists | Union-merge new fields into existing override's `fields` array |
| File covered by an **exact-path** pattern, with `--fields`, no override exists | Append `[[seal.override]]` block (exact path already in `patterns`; no duplicate appended) |
| File covered by an **exact-path** pattern, with `--fields`, override exists | Union-merge new fields into existing override's `fields` array |
| File covered exactly (or by wildcard), no `--fields`, no override | No config change (file is already fully covered) |
| File covered by `[[seal.exclude]]` | Warn to stderr and **proceed with in-place encryption**; do not modify config. (`[[seal.exclude]]` suppresses `status` drift reporting for this file, but does not prevent explicit `seal` invocations.) |
| File path starts with `.gitvault/` | Exit with error: `cannot seal gitvault configuration files` |

All config mutations use `toml_edit` to preserve comments and formatting in the existing file.

### Drift detection (`gitvault status` and `status --fix`)

```
gitvault status [OPTIONS]
```

| Flag | Default | Description |
|---|---|---|
| `--fail-if-dirty` | off | Exit with code 6 if any drift (store or seal) is found; used by CI and pre-push hook |
| `--fix` | off | Apply automatic repairs (gitignore entries, missing patterns). Mutually exclusive with `--staged` |
| `--json` | off | Output machine-readable JSON: `{"store":[{"path":"…","status":"ok\|drifted\|orphaned"}],"seal":[{"path":"…","status":"ok\|drifted\|excluded","fields":["…"]}]}` |
| `--staged` | off | Inspect only git-staged files (reads staged content via `git show :<path>`); skips store drift. For hook use. Mutually exclusive with `--fix` |
| `--sealed-only` | off | Skip store-drift check; report only seal drift |
| `-e, --env <ENV>` | `GITVAULT_ENV` / `dev` | Environment to use for store-drift resolution |

`status` (read-only by default) reports two categories unified in one output:

**Category 1 — Store drift** (existing REQ-32): `.age` files in `.gitvault/store/<env>/` whose source has changed.

**Category 2 — Seal drift** (new): evaluates each pattern in `[seal].patterns`:

1. Glob the working tree for matching files (respecting `[[seal.exclude]]` and always skipping `.gitvault/**`).
2. For each matched file, read it and identify encrypted values.
3. If a `[[seal.override]]` exists for this file, verify that **each** listed field is encrypted (not just one — all must be sealed).
4. If no override exists, verify at least one encrypted value exists anywhere in the file.
5. Drift = any expected field is found to be plaintext.

```
gitvault status

Store:
  ✓  prod/services/auth/config.json.age    (up to date)
  ✗  prod/.env.age                         drift: source modified
  ⚠  prod/config.json.age                 orphaned: no matching source file (pre-migration flat layout)

Sealed:
  ✓  helm/values.yaml                      (2/2 fields sealed)
  ✓  config/app.json                       (all fields sealed)
  ✗  .env                                  drift: 0 of 3 values encrypted — run 'gitvault seal .env'
     config/public.json                    excluded
```

Exit code 6 when `--fail-if-dirty` is set and any drift (store `drifted` or seal `drift`) is found. Orphaned store entries do **not** trigger exit code 6.

**`status --fix`** applies automatic repairs without re-sealing or re-encrypting:

| Repair | Action |
|---|---|
| Missing `.gitignore` entry for gitvault output dirs | Appends entry to `.gitignore` |
| File with encrypted values not covered by `[seal].patterns` | Appends exact-path entry to `patterns` |
| Override union / drift re-seal | **Not performed** — `--fix` is non-destructive to file content |

`--fix` reports each change on stderr. It never modifies file content, never adds `[[seal.override]]` blocks, and never removes patterns.

### Why not `sealed.toml`?

| Property | `sealed.toml` (state file) | `config.toml [seal]` (intent patterns) |
|---|---|---|
| Merge conflicts | Yes — two branches both write to same keys | No — pattern list is append-only |
| New files auto-covered | No — must run `seal` and commit index | Yes — wildcard patterns cover future files |
| Drift detection | State vs. disk comparison | Intent vs. disk comparison |
| Familiar mental model | Lockfile | `.gitignore` / `.gitattributes` |
| Requires `seal` to update index | Always | Only when file isn't already covered |
| Comment preservation | N/A | Required — use `toml_edit` for mutations |

## `src/config.rs` Schema Extension

The `GitvaultConfig` struct gains a `seal: SealConfig` field. Intermediate raw structs follow the existing `deny_unknown_fields` pattern so unknown `[seal]` sub-keys produce a usage error immediately:

```rust
pub struct GitvaultConfig {
    pub hooks: HooksConfig,
    pub env: EnvConfig,
    // ... existing fields ...
    pub seal: SealConfig,    // new
}

pub struct SealConfig {
    pub patterns: Vec<String>,
    pub overrides: Vec<SealOverride>,
    pub excludes: Vec<SealExclude>,
}

pub struct SealOverride {
    pub pattern: String,
    pub fields: Vec<String>,
}

pub struct SealExclude {
    pub pattern: String,
}
```

`SealConfig::default()` returns empty vectors (no patterns = no sealing configured). Absence of `[seal]` in `config.toml` is not an error. The `toml_edit` crate handles all writes; the `toml` + serde path continues for reads.

## Migration Path

```bash
# Before → After

gitvault encrypt app.json --fields db.pass --env dev
→ gitvault seal app.json --fields db.pass

gitvault encrypt .env --value-only --env dev
→ gitvault seal .env

gitvault decrypt .gitvault/store/dev/app.json.age --fields db.pass --reveal
→ gitvault unseal app.json --fields db.pass --reveal

gitvault decrypt .gitvault/store/dev/.env.age --value-only
→ gitvault unseal .env
```

## Test Plan

- Roundtrip: `seal` then `unseal --reveal` returns original content for JSON/YAML/TOML/.env.
- Idempotency: `seal` on already-sealed file produces identical ciphertext (REQ-5).
- `--fields` narrow: only listed fields change; others untouched.
- `unseal` in-place: plaintext written to file; both warnings emitted to stderr.
- `unseal` in-place on git-tracked file: git-tracked warning emitted.
- `unseal --reveal`: stdout contains plaintext; file on disk unchanged.
- Unsupported extension: usage error with guidance.
- `.gitvault/config.toml` as seal target: rejected with error.
- `.env --fields` with dot-path: rejected with usage error; top-level key works.
- **config.toml created from scratch:** `seal` on repo with no `config.toml` creates the file.
- **config.toml auto-update — exact path:** appended to `patterns`; subsequent `seal` does not re-append.
- **config.toml auto-update — with --fields:** appends pattern + `[[seal.override]]` block.
- **config.toml auto-update — wildcard covered, --fields:** appends only override block.
- **config.toml auto-update — override union merge:** existing `["api.key"]` + `--fields db.pass` → `["api.key", "db.pass"]`.
- **config.toml not modified:** already fully covered file → byte-identical config after seal.
- **config.toml comment preservation:** comments in existing `config.toml` survive mutation.
- **config.toml update failure:** manual fragment printed to stderr; sealed file not rolled back.
- **status unified output:** store drift + seal drift reported together.
- **status drift precision:** `[[seal.override]]` with 2 fields, 1 plaintext → reported as drift.
- **status clean:** after `seal`, all pattern-matched files reported as sealed.
- **status exclude:** `[[seal.exclude]]` file not reported as drift.
- **status --fix gitignore:** missing output dir entry added to `.gitignore`.
- **status --fix pattern:** file with encrypted values but no pattern entry gets exact-path added.
- **status --fix non-destructive:** override blocks untouched; file content untouched.
- `rekey` re-seals all pattern-matched files with new recipients; respects `[[seal.override]]` fields.
