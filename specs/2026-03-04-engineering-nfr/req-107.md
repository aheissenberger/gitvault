# REQ-107: Atomic File Write Utility

**ID:** S-20260304-R0107
**Status:** Implemented
**Priority:** Medium
**Category:** Engineering / Reliability

## Problem

Multiple call sites in the codebase independently implement a write-temp-then-rename pattern
using `tempfile::NamedTempFile`. When duplicated, each site risks divergent error handling,
missing fsync, or incorrect temp-dir placement (different filesystem than target), which would
cause the rename to fail or — worse — silently produce a non-atomic cross-device copy.

## Requirement

A single `fs_util::atomic_write(path: &Path, data: &[u8]) -> Result<(), GitvaultError>`
function SHALL be the canonical implementation for all file writes that must not leave partial
content on failure. All new file-write code MUST use this utility unless an intermediate step
(e.g., permission change) is required between the write and rename.

## Exceptions

Sites that require a permission-change or ownership step between write and rename (e.g.,
`barrier.rs` token/key files) SHALL document the exception with a
`// fs_util::atomic_write not used here: <reason>` comment.

## Acceptance Criteria

- AC1: `src/fs_util::atomic_write` exists and uses a temp file in the target directory.
- AC2: `src/structured/helpers.rs::atomic_write` delegates to `fs_util::atomic_write`.
- AC3: All non-excepted call sites use `fs_util::atomic_write`.
- AC4: Exception sites carry the documented comment explaining the exception.

## Implementation

- `src/fs_util.rs`: `pub fn atomic_write`
- `src/structured/helpers.rs`: delegating wrapper
- `src/ssm/refs.rs`: migrated
