# REQ-114: Breaking Changes to `decrypt`

**ID:** S-20260304-R114
**Status:** Proposed
**Priority:** High
**Category:** Breaking Change / DX Coherence

## Problem

`gitvault decrypt` currently requires the caller to know and provide the `.age` store path:

```bash
gitvault decrypt .gitvault/store/prod/config.json.age --reveal
```

This is an implementation detail that leaks the store's directory structure into the user's
workflow. It also means that scripts must reconstruct the store path manually, and users who
move files (changing the source path) must remember to update their decrypt commands.

Additionally, `decrypt` accepts `--fields` and `--value-only` which were already removed from
`encrypt` (REQ-113) and superseded by `seal`/`unseal` (REQ-112). Their presence on `decrypt`
creates a mismatch and must be removed for consistency.

## Breaking Changes

### 1. Source-path resolution — accept original path

Users now pass the **source file path**, not the store path:

```bash
# Before
gitvault decrypt .gitvault/store/prod/services/auth/config.json.age

# After
gitvault decrypt services/auth/config.json --env prod
```

The source path is resolved to the store path mechanically:
```
<source-path>  →  .gitvault/store/<env>/<source-path>.age
```

**Backward compatibility:** `.age` paths are still accepted. When `<FILE>` ends with `.age`,
it is used directly (no resolution). This allows existing CI scripts to continue working
during a migration period.

```bash
# Still works (explicit .age path)
gitvault decrypt .gitvault/store/prod/services/auth/config.json.age
```

### 2. Remove `--fields`

`decrypt --fields` is superseded by `unseal --fields` (REQ-112):

```bash
# Before
gitvault decrypt .gitvault/store/prod/config.json.age --fields db.password --reveal

# After
gitvault unseal config.json --fields db.password --reveal
```

### 3. Remove `--value-only`

`decrypt --value-only` is superseded by `unseal` on `.env` files:

```bash
# Before
gitvault decrypt .gitvault/store/dev/.env.age --value-only

# After
gitvault unseal .env
```

### 4. Remove REQ-110 auto-discover branch

The auto-discover branch added by REQ-110 (detecting non-`.age` JSON/YAML/TOML files and
auto-decrypting encrypted fields) is removed from `cmd_decrypt`. Its functionality is now
provided cleanly by `unseal`:

```bash
# REQ-110 auto-discover (removed)
gitvault decrypt config.json --reveal

# Replacement (REQ-112)
gitvault unseal config.json --reveal
```

The removal avoids ambiguity: `decrypt` always means "fetch from store". `unseal` always
means "decrypt fields in-place or reveal to stdout".

## `decrypt` Command After Changes

```
gitvault decrypt [OPTIONS] <FILE>
```

| Flag | Default | Description |
|---|---|---|
| `<FILE>` | — | Original source path (or explicit `.age` store path) |
| `--reveal` | off | Print decrypted content to stdout; do not write to `.git/gitvault/plain/` |
| `-e, --env <ENV>` | — | Environment for store path resolution (overrides `GITVAULT_ENV`) |
| `-i, --identity <IDENTITY>` | — | Identity key file |

**Removed flags:** `--fields`, `--value-only`

**Source-path resolution algorithm:**

```
if FILE ends with ".age":
    use FILE directly as store path
else:
    store_path = <repo-root>/.gitvault/store/<env>/<FILE>.age
    if store_path does not exist:
        error: "No encrypted archive found for '<FILE>' in environment '<ENV>'."
```

## Implementation Notes

- Add `resolve_store_path(source: &Path, env: &str, repo_root: &Path) -> PathBuf` to
  `src/commands/decrypt.rs` (or a shared `src/store.rs` module used by both encrypt and decrypt).
- The function is the inverse of `compute_output_path` from REQ-113.
- Remove `auto_discover` branch (added in REQ-110, lines ~138–180 in current `decrypt.rs`).
- Remove `--fields` and `--value-only` arms from the match in `cmd_decrypt`.
- Update CLI struct in `cli.rs`: remove `fields`, `value_only` from `Decrypt` variant.
- Update `dispatch.rs`: remove corresponding arguments from `cmd_decrypt` call.
- The `structured` module's `collect_encrypted_field_paths` and `decrypt_fields_content`
  remain in the codebase (used by `unseal`); they are no longer called from `cmd_decrypt`.
