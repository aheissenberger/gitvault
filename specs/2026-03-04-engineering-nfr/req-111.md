# REQ-111: Encrypt All String Fields Without Explicit Listing (`--all-fields`)

**ID:** S-20260304-R111
**Status:** Proposed
**Priority:** Medium
**Category:** Developer Experience / Usability

## Problem

`gitvault encrypt config/app.json --fields database.password,api.key` requires the operator
to enumerate every sensitive field. For files that are dedicated secrets stores (all values
are secrets), listing each field is repetitive and error-prone — a newly added field is
silently left in plaintext if the `--fields` list is not updated.

The symmetric `decrypt` command gained auto-discover in REQ-110. Encrypt needs the
complementary capability.

## Decision

Add `--all-fields` as a dedicated boolean flag to `gitvault encrypt`. When set, the command:

1. Recursively walks every string leaf in the document.
2. Collects their dot-path keys (same recursive visitor pattern as REQ-110).
3. Passes the full list to the existing `encrypt_fields` logic, which handles idempotency
   (REQ-5: already-encrypted values are kept unchanged).

`--all-fields` is intentionally a separate flag rather than a sentinel value for `--fields`
(e.g. `--fields '*'`) to avoid shell glob-expansion footguns when the `*` is left unquoted.

## New Public API (`src/structured`)

```rust
/// Recursively collect dot-path keys of every **string** leaf value in a JSON, YAML,
/// or TOML document — regardless of whether the value is already age-encrypted.
///
/// Non-string leaf types (numbers, booleans, null, arrays, objects) are skipped.
/// Returns an empty `Vec` when no string leaves are found.
///
/// # Errors
///
/// Returns [`GitvaultError`] if `content` cannot be parsed for the given `ext`.
pub fn collect_all_string_field_paths(
    content: &str,
    ext: &str,
) -> Result<Vec<String>, GitvaultError>;
```

## CLI Changes

### `src/cli.rs` — `Encrypt` variant

```rust
/// Encrypt all string fields in a JSON/YAML/TOML file without specifying each one
/// (conflicts with --fields; use --value-only for .env files)
#[arg(long, conflicts_with = "fields")]
all_fields: bool,
```

### `src/commands/encrypt.rs` — new branch in `cmd_encrypt`

Inserted between the `--fields` branch and the `--value-only` branch:

```
encrypt <FILE> [options]
├── .age input?           → reject (REQ-33: no re-encryption of .age blobs)
├── --fields given        → existing explicit field-encrypt path (unchanged)
├── --all-fields given
│   ├── ext is .json/.yaml/.yml/.toml
│   │   ├── collect_all_string_field_paths(content, ext)
│   │   ├── if empty → notice + Ok
│   │   └── encrypt_fields(path, &paths, identity, recipient_keys)
│   └── else → UsageError("--all-fields requires JSON/YAML/TOML; use --value-only for .env")
├── --value-only          → existing .env per-value path (unchanged)
└── whole-file age stream → unchanged
```

## Design Choices

| Choice | Rationale |
|---|---|
| Separate `--all-fields` flag (not `--fields '*'`) | Shell glob expansion makes `--fields *` a footgun; a dedicated flag is unambiguous |
| `conflicts_with = "fields"` | Combining explicit paths and "all" is semantically redundant and error-prone |
| Reuse `encrypt_fields` for execution | REQ-5 idempotency is already implemented there; no duplication |
| Skip non-string leaves silently | Numbers, booleans, ports are never valid age ciphertext targets; encrypting them would corrupt structured data |
| Reject `.env` with usage error | `.env` value-level encryption already has a dedicated `--value-only` path; overlap would be confusing |
| `collect_all_string_field_paths` as a new function | Distinct from `collect_encrypted_field_paths` (REQ-110) which filters only already-encrypted values; the encrypt variant must collect all strings |

## Relationship to REQ-110

| | REQ-110 (decrypt auto-discover) | REQ-111 (encrypt --all-fields) |
|---|---|---|
| Direction | Decrypt → plaintext | Plaintext → encrypt |
| Trigger | Implicit (no `--fields`) | Explicit (`--all-fields` flag) |
| Filter | Only age-armored values | All string leaf values |
| Error on failure | Skip-with-warning (per field) | Fail fast (encryption never partially fails) |
| New function | `collect_encrypted_field_paths` | `collect_all_string_field_paths` |

The decrypt side uses **implicit** auto-discover (safe: worst case is "nothing to do") while
the encrypt side uses an **explicit** flag (safer default: whole-file encryption remains the
default to avoid accidentally encrypting non-sensitive structured data like version strings,
hostnames, or feature flag names).

## Test Plan

- Unit: `collect_all_string_field_paths` returns all string leaves in JSON/YAML/TOML.
- Unit: nested objects and arrays are walked; index paths (`servers.0.token`) are returned.
- Unit: non-string leaves (numbers, booleans) are excluded.
- Unit: REQ-5 re-run: encrypt → encrypt again → ciphertext unchanged.
- Unit: `--all-fields` on `.txt` extension returns a usage error.
- Integration: `cmd_encrypt --all-fields` on a partial-plaintext JSON file encrypts only the
  previously unencrypted string fields; already-encrypted fields are left unchanged.
