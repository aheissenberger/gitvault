# REQ-115: `gitvault edit` — Open Sealed or Encrypted Files in an Editor

## Context

Developers frequently need to update a single secret inside a sealed JSON/YAML/TOML/env
file or inside a `.age`-encrypted store archive. The current workflow requires three manual
steps: unseal (or decrypt), edit, re-seal (or re-encrypt). A single `gitvault edit` command
eliminates this friction while keeping plaintext off disk as much as possible.

## Goal

`gitvault edit <FILE>` provides a safe, single-command workflow for editing secrets:

- **Sealed-file mode** (`.json`, `.yaml`, `.yml`, `.toml`, `.env`): unseal values to a temp
  file, open in editor, re-seal on save.
- **Store-file mode** (`.age` or source path resolving to a store archive): decrypt to a
  temp file, open in editor, re-encrypt on save.

## Non-goals

- Shell-level key-by-key editing (use `gitvault seal --fields` for targeted sealing).
- Creating new files (use `gitvault encrypt` or `gitvault seal`).
- GUI editor integration beyond launching the process and waiting for exit.

## Constraints

- Plaintext MUST NOT be written inside the repository working tree.
- The temp file MUST be zeroized before deletion.
- The `zeroize` crate is already a project dependency — no new dependencies.
- `std::fs::canonicalize` MUST NOT be used for store path computation (use lexical
  normalization, same as REQ-113/114).

## Acceptance Criteria

See frontmatter `acceptance` list (AC1–AC10).

## Test Plan

- Unit tests for `resolve_editor()` covering all five priority levels.
- Integration test (sealed-file mode): seal a `.env` file, run `cmd_edit` with `true` as
  editor (no-op), verify file unchanged and "No changes" reported.
- Integration test (store-file mode): encrypt a file to store, run `cmd_edit` with a script
  that appends a line, verify the store `.age` file changed and decrypts correctly.
- Integration test: `--fields` in store-file mode returns usage error.

## Notes

- On macOS, `open -W -n <file>` waits for the specific window to close. Without `-W`,
  the process returns immediately; without `-n`, `open` may reuse an existing window and
  return before the user has saved.
- On Windows, `notepad.exe` returns immediately when invoked programmatically. Users who
  need a blocking editor should set `[editor] command = "code --wait"` in config.
- The `--env` asymmetry between sealed-file mode (recipients only) and store-file mode
  (path + recipients) mirrors the existing asymmetry between `seal` (REQ-112 AC5) and
  `encrypt`/`decrypt` (REQ-113/114).
