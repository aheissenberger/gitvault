## Requirement
- REQ-68

## Context
REQ-67 introduced repository-level canonical config (`.gitvault/config.toml`) for hook adapter selection. Teams also need optional user-level defaults for local ergonomics (for example default secret folder naming and plugin defaults) without weakening repository policy or CI determinism.

## Config Layers
1. CLI arguments and environment variables
2. Repository config: `.gitvault/config.toml`
3. User-global config: `~/.config/gitvault/config.toml` (optional)
4. Built-in defaults

## Rules
- Repository config remains canonical for project policy and collaboration.
- User-global config is optional and only supplies fallback values when repo config does not set a key.
- Unknown keys in known sections, malformed TOML, and invalid values are `Usage` errors in both files.
- If user-global file is absent, gitvault behaves exactly as before.
- If user-home cannot be resolved/accessed, global layer is skipped and resolution continues deterministically.
- This requirement defines layering/validation behavior only; semantic keys are defined by their owning requirements.
- Any key that would violate worktree isolation or fail-closed security defaults is out of scope for global overrides.

## Verification
- Unit tests: no `~/.config/gitvault/config.toml` -> behavior unchanged from existing defaults.
- Unit tests: global-only valid config -> value is applied.
- Unit tests: same key in global and repo config -> repo value wins.
- Unit tests: unknown key in global or repo config -> `GitvaultError::Usage`.
- Unit tests: malformed TOML in global or repo config -> `GitvaultError::Usage`.
- Unit tests: HOME unavailable/unreadable global path -> deterministic fallback to repo/default, no interactive branch.
- Integration tests: CI/`--no-prompt` paths remain deterministic under all layer combinations.

## Implementation-facing Acceptance Scenarios
- R068-S1: Given user-global config does not exist, when commands run, then behavior is identical to current defaults and repo config handling.
- R068-S2: Given user-global config exists and repo config lacks the key, when command runs, then the global value is used.
- R068-S3: Given both user-global and repo config set the same key, when command runs, then repo value is selected.
- R068-S4: Given CLI/env sets a value and both config files also set it, when command runs, then CLI/env value wins.
- R068-S5: Given unknown key or invalid value in either config file, when parsing occurs, then command fails with `Usage` and actionable message.
- R068-S6: Given multiple parallel worktrees, when commands run in each worktree, then effective configuration is resolved deterministically per precedence without cross-worktree contamination.
- R068-S7: Given user-home resolution is unavailable, when command runs, then global config is skipped and deterministic repository/default behavior is used.
- R068-S8: Given malformed TOML in either config layer, when parsing occurs, then command fails with `Usage` and actionable diagnostics.
- R068-S9: Given optional adapter key is set to empty string and no higher-precedence non-empty value exists, when command runs, then value is treated as unset.
