## Requirement
REQ-86

## Context
`now_secs()` in `src/barrier.rs` currently handles a clock-before-epoch error silently:

```rust
fn now_secs() -> u64 {
    SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .unwrap_or_else(|_| std::time::Duration::from_secs(u64::MAX))
        .as_secs()
}
```

When the system clock is set before January 1, 1970 (e.g., a misconfigured VM, a system clock reset, or an adversarial environment), `duration_since` returns `Err`. The current code silently returns `u64::MAX`.

The downstream effects:
- `has_valid_token()`: `u64::MAX < expiry` is always false for any realistic expiry, so all tokens appear expired. This is fail-closed (safe), but produces no actionable error message — the user sees only "no valid prod allow token" without understanding why.
- `allow_prod()`: `u64::MAX.checked_add(ttl_secs)` returns `None` → `Err("timestamp overflow")`. The error message does not mention the clock, making it confusing to diagnose.

The silent `u64::MAX` substitution is a smell: a clock that is decades wrong should produce a loud, actionable error, not a silent integer substitution.

## Goal
Convert the silent clock-error swallow into an explicit, actionable error that surfaces to the user with a clear diagnostic message.

## Non-goals
- Correcting the system clock.
- Retrying with NTP synchronization.
- Supporting pre-1970 token expiry times (nonsensical use case).

## Constraints
- `now_secs` is a private function; its signature change is an internal refactor only.
- `has_valid_token` must remain a `-> bool` function (callers depend on this); it absorbs the `Result` and logs the warning.
- `allow_prod` already returns `Result<u64, GitvaultError>` and can propagate the error cleanly.
- The warning in `has_valid_token` must go to stderr (not stdout or log files).

## Acceptance Criteria
- AC1: `now_secs()` returns `Result<u64, GitvaultError>`; clock error → `Err(Other(...))`.
- AC2: `has_valid_token` absorbs the error, returns `false`, logs warning to stderr.
- AC3: `allow_prod` propagates the `now_secs` error.
- AC4: No `.unwrap()` or `.expect()` on `now_secs()` result.
- AC5: Tests for clock-before-epoch scenario; all existing tests pass.

## Test Plan
- Existing `allow_prod_with_overflow_ttl_returns_error` test exercises the `checked_add` path; must still pass.
- New test: inject a `now_secs` that returns `Err` → `has_valid_token` returns `false`.
- New test: inject a `now_secs` that returns `Err` → `allow_prod` returns `Err(Other(...))`.
- `now_secs_returns_reasonable_timestamp` test must still pass (clock is valid in CI).

## Notes
The current behaviour (return `u64::MAX`) was likely chosen to keep the function simple and infallible. The right approach is the explicitly-fallible version, since the alternative (silently substituting an extreme value) masks a real environmental problem. This change also eliminates one more implicit `unwrap_or_else` that hides errors — consistent with the codebase's general move toward explicit error handling.
