## Requirement

REQ-70 — `gitvault harden` file import

### Rationale

Extending `harden` to accept file arguments provides a clean, semantically coherent migration
path for teams moving from plaintext committed secrets to gitvault. The word "harden" already
means "make this safer" at the repository level; applying the same verb to individual files is
natural and requires no new concept.

The alternative (a dedicated `import` subcommand or an `--import` flag on `encrypt`) was
rejected because:
- A separate `import` command splits the "make repo safe" workflow across two verbs.
- An `--import` flag on `encrypt` conflates a pure cryptographic operation with a git state
  mutation, and is easy to forget.

### CLI design

```
# Harden repository only (existing behaviour, unchanged):
gitvault harden

# Import one file:
gitvault harden .env

# Import multiple files / glob:
gitvault harden .env config/*.env secrets-backup.json

# Import with explicit environment:
gitvault harden .env --env staging

# Import with field-level encryption:
gitvault harden config.json --fields db.password,api_key

# Preview without making changes:
gitvault harden .env --dry-run

# Also delete source after encryption:
gitvault harden .env --remove

# Non-interactive (CI):
gitvault harden .env --no-prompt
```

### Per-file import sequence

```
for each resolved file:
  1. Skip if already imported (age file exists + source untracked + source gitignored)
  2. Encrypt → secrets/<env>/<file>.age  (via existing encrypt logic)
  3. Verify age output is valid
  4. git rm --cached <file>             (only if tracked)
  5. Append to .gitignore               (only if not already covered)
  6. rm <file>                          (only if --remove passed)
```
