## Requirement
REQ-82

## Context
`escape_env_value` in `src/materialize.rs` (line 83) escapes backslash, double-quote, `$`, `\n`, and `\r` for embedding values in double-quoted `.env` lines. Two characters are missing:

**Backtick (`` ` ``):** In double-quoted bash strings, `` `command` `` and `$(command)` are both evaluated as command substitutions. `$` is already escaped (preventing `$()`), but backtick substitution is independent and unescaped. If a secret value contains a backtick (e.g. a password like `` pass`word ``), and the `.env` file is `source`'d or `eval`'d by a bash script, the backtick triggers arbitrary command execution with the permissions of the sourcing script. This is a command injection vector.

**NUL byte (`\0`):** POSIX shell variables cannot contain NUL bytes; most `.env` parsers truncate at NUL. A NUL byte in a secret value will silently corrupt the output, causing the environment variable to be shorter than the actual secret. This is a data integrity failure that could also be exploited (e.g., a secret `password\0evil` appears to be `password` in the `.env` but is stored fully in the encrypted file).

## Goal
Close the command injection vector (backtick) and the data-truncation vector (NUL) in `.env` file generation.

## Non-goals
- Escaping characters that are safe in double-quoted bash strings (e.g., `!`, `?`, `*`).
- Handling `$(...)` (already handled by `$` escaping).
- Modifying the `.env` parser or reader paths.

## Constraints
- The fix is additive to the existing `match` statement — add two new arms.
- NUL replacement must be lossy (NUL → `\0` literal); no error is raised, only a warning.
- Backtick escape must produce `\`` (two bytes) which bash interprets as a literal backtick.
- The doc comment must be updated to keep the escape table accurate.

## Acceptance Criteria
- AC1: Backtick escaped as `` \` `` in output.
- AC2: NUL replaced with `\0` literal; warning emitted to stderr.
- AC3: Existing escape behaviour unchanged.
- AC4: Doc comment updated with complete escape table.
- AC5: New unit tests for backtick-only, mixed backtick, NUL-only, multi-NUL, and combined cases.

## Test Plan
- Unit test: `escape_env_value("` + "`" + `")` → `"\\` + "`" + `"`.
- Unit test: `escape_env_value("a` + "`" + `b")` → `"a\\` + "`" + `b"`.
- Unit test: `escape_env_value("a\0b")` → `"a\\0b"` (and stderr warning captured).
- Unit test: `escape_env_value("\0\0")` → `"\\0\\0"`.
- Unit test: `escape_env_value("` + "`" + `\0")` → `"\\` + "`" + `\\0"`.
- Integration test: materialize a secret with a backtick in its value; verify `.env` output contains `` \` ``.

## Notes
The backtick fix is the higher-severity item (code injection). The NUL fix is data-integrity. Both are two-line additions to the existing `match` block. This is the fastest fix in the set after REQ-80.
