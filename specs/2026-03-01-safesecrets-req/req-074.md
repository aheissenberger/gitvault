## Requirement
REQ-74

## Context
The most ergonomic and secure way to pass a secret to a CLI tool in a pipeline is via stdin. The key flows through a kernel pipe buffer — it is never in any env var, never in `/proc/<PID>/environ`, and never in `ps` output. This is the pattern used by `docker login --password-stdin`, `gpg --passphrase-fd 0`, `age --identity -`, and similar tools. Adding `--identity-stdin` makes `gitvault` first-class in pipeline compositions.

## Goal
Allow callers to pipe an age private key directly to `gitvault` without exposing the key material in any process-visible location. This is the recommended pattern for GitHub Actions (pipe from `/dev/shm` file or from a secrets manager CLI) and for any shell script that already holds the key in a variable.

## Non-goals
- Interactive passphrase prompting via stdin (that is handled by `GITVAULT_IDENTITY_PASSPHRASE` and the existing passphrase callback chain).
- Reading multiple identities from stdin.
- Replacing `GITVAULT_IDENTITY` or `GITVAULT_IDENTITY_FD` (all three sources remain available).

## Constraints
- stdin must not be consumed unless `--identity-stdin` is explicitly set.
- The flag must be consistently wired to every command that accepts `--identity`.
- TTY detection (`isatty(0)`) must be performed before attempting to read; hanging on a terminal is never acceptable.
- Key material must be placed in `Zeroizing<String>` immediately after reading.

## Acceptance Criteria
- AC1: Stdin read produces correct identity; whitespace is stripped; key is zeroized on drop.
- AC2: Conflict with `--identity` produces exit code 2 immediately.
- AC3: Priority is identical to `--identity` (explicit flag, highest precedence).
- AC4: TTY stdin → exit code 2 with actionable message (no hang).
- AC5: `--json` mode unaffected.
- AC6: `check` command probes stdin without hanging on TTY.

## Test Plan
- Unit test: pipe a known key via `stdin` redirect; confirm correct identity loaded and key zeroized.
- Unit test: supply both `--identity` and `--identity-stdin`; confirm exit code 2.
- Unit test: `isatty` mock returns true; confirm exit code 2 and message.
- Integration test (all platforms): `printf '%s' "$KEY" | gitvault decrypt --identity-stdin` roundtrip.
- Integration test: `gitvault check --identity-stdin < /dev/null` exits non-zero with clear error (empty input).

## Usage Examples

```bash
# GitHub Actions — key written to /dev/shm, piped to avoid env var in child
printf '%s' "$AGE_KEY" | gitvault decrypt --identity-stdin

# From a secrets manager CLI
op read "op://vault/gitvault/age_key" | gitvault materialize --identity-stdin

# From a file (alternative to --identity <path>)
gitvault run --identity-stdin < ~/.config/gitvault/identity.age -- my-server
```

## Notes
This is the highest-value security improvement for GitHub Actions CI usage because it eliminates the need to store the key anywhere other than the pipe buffer. Documented CI recipe (see `29-identity-security-improvements.md`):

```yaml
- name: Decrypt secrets
  env:
    AGE_KEY: ${{ secrets.AGE_IDENTITY_KEY }}
  run: printf '%s' "$AGE_KEY" | gitvault decrypt --identity-stdin
```

The key is still briefly in `$AGE_KEY` (step-scoped env var), but it is not passed to the child `gitvault` process as an env var — only as a pipe.
