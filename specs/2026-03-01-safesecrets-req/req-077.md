## Requirement
REQ-77

## Context
`GITVAULT_IDENTITY_PASSPHRASE` always contains raw passphrase material — unlike `GITVAULT_IDENTITY`, there is no "safe" file-path form. Every use of this env var exposes the passphrase in `/proc/<PID>/environ` (Linux), `ps auxe` output, and audit logs that capture process environments. Users who rely on this env var in CI may be unaware of the exposure or unaware that secure alternatives exist: the OS keyring (already implemented), `GITVAULT_IDENTITY_PASSPHRASE_FD` (REQ-76, Unix), or simply using a native age key (no passphrase required).

The warning is unconditional on non-empty values (unlike REQ-75 for `GITVAULT_IDENTITY`, where file paths are safe). This is correct because no value of `GITVAULT_IDENTITY_PASSPHRASE` is non-sensitive.

## Goal
Alert users to the process-environment exposure risk at runtime, and direct them to the three secure alternatives, without removing backward compatibility.

## Non-goals
- Removing `GITVAULT_IDENTITY_PASSPHRASE` support.
- Changing exit codes or blocking execution.
- Warning about keyring-stored passphrases (they are not in env vars).
- Covering `GITVAULT_IDENTITY` inline key warning (that is REQ-75).

## Constraints
- Warning must go to stderr only; never stdout or structured JSON output.
- Warning detection fires for any non-empty value of `GITVAULT_IDENTITY_PASSPHRASE`.
- `GITVAULT_NO_PASSPHRASE_WARN=1` suppression is exclusively for test infrastructure.
- The warning must list the OS keyring as the first recommended alternative, since it is the most secure and already available on all platforms.

## Acceptance Criteria
- AC1: Warning emitted to stderr for any non-empty `GITVAULT_IDENTITY_PASSPHRASE`; alternatives listed.
- AC2: Warning suppressed in `--json` mode; emitted at most once per invocation.
- AC3: Passphrase still resolves correctly (no regression).
- AC4: Warning text is stable and matches the documented format.
- AC5: Test suppression env var works; test suite produces no spurious warnings.

## Test Plan
- Unit test: `GITVAULT_IDENTITY_PASSPHRASE=secret` → stderr contains `[gitvault WARN]` line; exit 0.
- Unit test: `GITVAULT_IDENTITY_PASSPHRASE=secret --json` → no `[gitvault WARN]` on stderr.
- Unit test: `GITVAULT_NO_PASSPHRASE_WARN=1 GITVAULT_IDENTITY_PASSPHRASE=secret` → no warning.
- Unit test: `GITVAULT_IDENTITY_PASSPHRASE` unset → no warning.
- Regression: existing passphrase tests pass without stderr noise (use suppression env var in test setup).

## Notes
The full priority list of secure alternatives the warning should surface:
1. **OS keyring** (`gitvault keyring set-passphrase`) — available on all platforms, no env var needed, passphrase stored securely in platform credential store. Already implemented.
2. **`GITVAULT_IDENTITY_PASSPHRASE_FD`** (Unix only, REQ-76) — passphrase in pipe buffer, not in env.
3. **Native age identity** — age native keys (`AGE-SECRET-KEY-…`) have no passphrase; eliminating the SSH-with-passphrase pattern eliminates the problem entirely.

This warning is the lowest-effort hardening step for the passphrase exposure surface. It should be implemented before or alongside REQ-76, and coordinated with the `GITVAULT_IDENTITY` inline warning (REQ-75) so both warnings share consistent format and suppression mechanics.
