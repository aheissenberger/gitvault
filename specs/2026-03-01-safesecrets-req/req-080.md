## Requirement
REQ-80

## Context
`atomic_write` in `src/structured/helpers.rs` contains:

```rust
tmp.write_all(data)
    .expect("NamedTempFile write_all is infallible in normal operation");
```

The comment's claim is incorrect: `write_all` on a `NamedTempFile` can fail on:
- Disk full (`ENOSPC`)
- I/O errors on network filesystems
- Quota exceeded (`EDQUOT`)
- Read-only filesystem mounted mid-operation

Any of these would cause a `panic!` in production code, terminating the process without a clean error message or proper exit code. Library code must never panic on recoverable errors (Rust API Guidelines). `GitvaultError::Io` already exists and is the correct return type for this failure.

## Goal
Convert a panicking code path in production to a proper `Result` propagation, consistent with all other fallible operations in the codebase.

## Non-goals
- Adding retry logic for transient I/O failures.
- Changing `atomic_write`'s signature (it already returns `Result<(), GitvaultError>`).
- Refactoring other functions.

## Constraints
- This is a one-line change. No scope creep.
- The fix must not regress any existing test.

## Acceptance Criteria
- AC1: `write_all` error propagated via `?`.
- AC2: No remaining `expect()`/`unwrap()` in production paths in `helpers.rs`.
- AC3: Existing tests pass; new test for `write_all` failure branch added.
- AC4: Exactly one line changed in the implementation.

## Test Plan
- Existing tests: pass unchanged.
- New test: write a `NamedTempFile` to a path on a read-only filesystem (or use a `Write` mock that returns `Err`) to cover the new `?` branch and confirm `GitvaultError::Io` is returned (not a panic).

## Notes
This is the simplest fix in the security hardening set (🟠 MEDIUM, one line). It should be implemented immediately — it requires no design decisions and reduces the panic surface of library code. `cargo clippy -D warnings` with `clippy::expect_used` lint would also catch this.
