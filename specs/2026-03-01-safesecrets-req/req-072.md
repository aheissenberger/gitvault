    text: "`gitvault rekey` is a new top-level command.   - id: "AC4"
    text: "Rekey uses a two-phase strategy. Phase 1 classifies every `.age` file: (a) **accessible** — decrypted into memory; (b) **no-access** — age returns a `no matching recipient` error, recorded as `skipped`; (c) **error** — any other failure (IO, corrupt ciphertext, parse error), which aborts the entire run immediately with exit code 1 and no files are modified. Phase 2 is reached only when there are no category-c errors: it writes all accessible files atomically (temp file + rename per file). Files in category (b) are not touched. This ensures the repository never ends up in a split-key state for files the caller has access to; files the caller cannot read are left unchanged."
  - id: "AC5"
    text: "Rekey always processes every `.age` file found; there is no skip-if-unchanged optimisation. The age format uses ephemeral X25519 keys, so the recipient public keys that were used to encrypt a file are not recoverable from the ciphertext — it is not possible to determine whether a file's current recipients match the desired set without trial decryption. Implementors must not attempt to infer equivalence from ciphertext. Every rekeyed file will differ from its predecessor (new ephemeral session key) and will appear as a git change; this is expected and correct."

## Context

This spec covers the pieces that complete **Option A — recipient ceremony** (zero-shared-secret team onboarding):

1. **`gitvault identity pubkey`** — print own public key (low-level escape hatch).
2. **`gitvault recipient add-self`** — add own public key to the recipients directory in one command (preferred path).
3. **`gitvault identity create --add-recipient`** — create key and register as recipient in a single step.
4. **Named per-person recipient files** — `.secrets/recipients/<name>.pub` for surgical git offboarding.
5. **`gitvault rekey`** — atomically re-encrypt the entire secrets tree to the current recipient set; **replaces and removes `gitvault rotate`** (breaking change).
6. **`defaults::RECIPIENTS_DIR` / `[paths] recipients_dir`** — constant and config key rename.

Together they complete the git-native, no-shared-secret onboarding/offboarding loop.

## Recipient ceremony flow

### Onboarding a new team member

```
# New member — on their machine (single command)
gitvault identity create --add-recipient
# → creates key, stores in keyring
# → writes .secrets/recipients/alice.pub (git username auto-detected)
git commit -m "onboard: add alice as recipient" && git push   # open PR

# Maintainer — after PR merges
git pull
gitvault rekey                        # re-encrypts all .secrets/**/*.age to new recipient set
git add .secrets/
git commit -m "rekey: add alice"
git push

# New member — after maintainer push
git pull
gitvault decrypt .secrets/dev/app.env.age   # works immediately
```

### Offboarding a departing member

```
# Maintainer — surgical removal via named file
git rm .secrets/recipients/dave.pub
git commit -m "remove dave as recipient"
gitvault rekey                        # re-encrypts; dave's key no longer opens any file
git add .secrets/
git commit -m "rekey: remove dave"
git push
```

> **Note:** `rekey` mitigates *future access*. To close the historical window (Dave may have copied plaintext), rotate the credentials at the provider level as a separate step.

### Recipients directory layout

```
.secrets/
  recipients/
    alice.pub      ← added by alice via identity create --add-recipient
    bob.pub        ← added by bob
    ci-bot.pub     ← added by maintainer: gitvault recipient add <key> --name ci-bot
  dev/
    app.env.age
  prod/
    app.env.age
```
