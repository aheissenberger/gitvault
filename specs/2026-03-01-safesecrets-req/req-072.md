## Context

This spec covers the two missing pieces that complete **Option A — recipient ceremony** (zero-shared-secret team onboarding):

1. **`gitvault identity pubkey`** — lets a developer print their own public key for inclusion in a PR without knowing where their key is stored.
2. **`gitvault rekey`** — lets a maintainer atomically re-encrypt the entire secrets tree to a new recipient set after a membership change.

Together they complete the git-native, no-shared-secret onboarding/offboarding loop.

## Recipient ceremony flow

### Onboarding a new team member

```
# New member — on their machine
gitvault identity create              # creates key, stores in keyring
gitvault identity pubkey              # prints: age1qqq...
# Copy output → open PR adding to .secrets/recipients/alice.pub

# Maintainer — after PR merges
git pull
gitvault rekey                        # re-encrypts all .secrets/**/*.age to new recipient set
git add .secrets/
git commit -m "rekey: add alice as recipient"
git push

# New member — after maintainer push
git pull
gitvault decrypt secrets/dev/app.env.age   # works immediately
```

### Offboarding a departing member

```
# Maintainer
git rm .secrets/recipients/dave.pub
git commit -m "remove dave as recipient"
gitvault rekey                        # re-encrypts; dave's key no longer opens any file
git add .secrets/
git commit -m "rekey: remove dave"
git push
```

> **Note:** `rekey` mitigates the *future access* risk. To mitigate the risk of Dave having already copied plaintext values, rotate the underlying credentials at the provider level as a separate step.
