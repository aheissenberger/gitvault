## Requirement
REQ-79

## Context
`allow_prod()` currently writes only `expiry.to_string()` to `.git/gitvault/.prod-token`. Any process with write access to that path can forge an arbitrary expiry timestamp and gain permanent prod access without going through the confirmation flow. While requiring write access to `.git/gitvault/` is already a meaningful prerequisite, defense-in-depth calls for making the token tamper-evident.

The token file is inherently a local-machine artifact (it only grants access on the machine where it was created). The HMAC key does not need to be shared across machines or kept across keyring resets — it is simply a local-machine secret that binds the token to the issuing machine.

## Goal
Make the allow token tamper-evident: an attacker who can write to `.git/gitvault/` but cannot read `.git/gitvault/.token-key` cannot forge a valid token with an extended expiry.

## Non-goals
- Cross-machine token portability (tokens are always machine-local).
- Token revocation beyond the existing `revoke_prod` command.
- Protecting against an attacker who can read AND write `.git/gitvault/` (they can read the key and forge tokens).
- Encrypting the token (HMAC authentication is sufficient; confidentiality is not required).

## Constraints
- The token file format change is a breaking change for any existing `.prod-token` files. Legacy bare-timestamp tokens must be treated as invalid (not as a security bypass).
- Constant-time HMAC comparison is mandatory to prevent timing-based forgery detection.
- The `.token-key` file must have the same `enforce_owner_rw` enforcement as the token file itself.
- The new crate dependencies (`hmac`, `sha2`, `subtle`) must be pinned to specific minor versions in `Cargo.toml`.

## Acceptance Criteria
- AC1: Token format is `<expiry>:<hex_hmac>`.
- AC2: HMAC key stored in `0600`-protected `.git/gitvault/.token-key`; generated once, reused.
- AC3: Verification re-computes HMAC and uses constant-time comparison; rejects on mismatch.
- AC4: HMAC failure treated as absent token; logged at debug level only.
- AC5: `.token-key` atomic write with permissions before rename.
- AC6: `revoke_prod` preserves `.token-key`.
- AC7: `hmac`, `sha2`, `subtle` added to `Cargo.toml`.

## Test Plan
- Unit test: `allow_prod` → token file contains `<digits>:<hex64>`.
- Unit test: `has_valid_token` accepts a correctly-signed future token.
- Unit test: `has_valid_token` rejects a token with a modified expiry (HMAC mismatch).
- Unit test: `has_valid_token` rejects a legacy bare-timestamp token.
- Unit test: `has_valid_token` rejects a token with a valid HMAC but expired timestamp.
- Unit test: verify `.token-key` has `0600` permissions on Unix.
- Unit test: `revoke_prod` leaves `.token-key` intact.

## Notes
The HMAC key threat model: the key protects against modification of the token file by a process that has write-but-not-read access to `.git/gitvault/` (e.g., a malicious git hook that writes blindly to that path). It does not protect against a process that can read the key file. Given that `.git/gitvault/` is owner-only (`0700` on Unix), this is an acceptable security model. The primary value is auditability: a tampered token will fail HMAC verification, producing an audit-detectable event.
