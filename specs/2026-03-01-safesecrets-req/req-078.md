## Requirement
REQ-78

## Context
`decrypt_armor()` and `decrypt_binary_b64()` in `src/structured/armor.rs` currently return `Vec<u8>`. A `Vec<u8>` is a heap allocation that Rust's allocator will free but will **not** overwrite with zeros. Until the allocator reuses the memory, the decrypted secret bytes remain readable on the heap — visible to:
- OS core dumps
- `/proc/<PID>/mem` readers (with sufficient privilege)
- Heap spray attacks
- Debugging tools attached to the process

The `zeroize` crate is already a direct dependency (via `zeroize = { version = "1", features = ["zeroize_derive"] }` in `Cargo.toml`). `Zeroizing<Vec<u8>>` is a drop-in wrapper that calls `Vec::zeroize()` (overwrites all bytes with 0x00) in its `Drop` implementation.

## Goal
Ensure that decrypted plaintext bytes are reliably overwritten with zeros when the value is dropped, minimizing the window during which secrets are readable in process heap memory.

## Non-goals
- Zeroizing ciphertext buffers (ciphertext is not sensitive).
- Preventing OS-level memory inspection by a privileged adversary (mlock/mprotect is out of scope here).
- Changing the behavior of `encrypt_armor` or `encrypt_binary_b64`.

## Constraints
- The `zeroize` crate must not be upgraded as part of this change.
- All existing tests must pass.
- The change is a type-level change only; no new error cases are introduced.

## Acceptance Criteria
- AC1: `decrypt_armor` returns `Zeroizing<Vec<u8>>`.
- AC2: `decrypt_binary_b64` returns `Zeroizing<Vec<u8>>`.
- AC3: All callers updated; no plain `Vec<u8>` alias of the decrypted data created.
- AC4: Intermediate `plaintext` buffer inside each function is `Zeroizing<Vec<u8>>`.
- AC5: No new crate dependencies.
- AC6: All existing tests pass.

## Test Plan
- Type-level check: `cargo build --all` succeeds (type mismatch would be compile error).
- Existing armor roundtrip tests pass.
- Optional: add a test that calls `decrypt_armor`, drops the result, and checks via `std::mem::ManuallyDrop` that the backing storage was zeroed — though this is platform-dependent and may be too fragile for CI.

## Notes
This is the highest-priority item in the security hardening set (🔴 HIGH). It requires coordinated updates to all callers (primarily in `src/structured/fields.rs`) and should be implemented first since subsequent changes in fields.rs will need to handle `Zeroizing<Vec<u8>>` anyway.
