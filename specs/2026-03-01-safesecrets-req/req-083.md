## Requirement
REQ-83

## Context
The `age` encryption format includes one encrypted stanza in the header per recipient. With no limit on `recipient_keys.len()`:

1. **Memory DoS**: constructing a `Vec<Box<dyn Recipient + Send>>` with thousands of entries allocates O(n) heap memory before any write.
2. **CPU DoS**: each stanza requires one X25519 key exchange; 100,000 recipients → 100,000 ephemeral key exchanges.
3. **File size DoS**: each stanza is ~130 bytes; 100,000 recipients → ~13 MB of header on a file that might encrypt 1 byte of payload.
4. **Parser DoS**: a maliciously crafted `.age` file with a huge recipient list could slow down any tool that reads the header.

In practice, no legitimate team has more than a few hundred members. A limit of 256 covers the 99.9th percentile of legitimate use while bounding worst-case resource consumption to manageable levels.

## Goal
Prevent memory, CPU, and storage exhaustion attacks through unbounded recipient lists by enforcing a configurable upper bound validated before any cryptographic work begins.

## Non-goals
- Rate limiting encrypt operations (out of scope).
- Per-environment recipient limits (the global limit is sufficient).
- Changing the recipient storage format.

## Constraints
- The validation must occur at the entry point of each encrypt function, before the `recipients` vector is populated.
- The constant must be in `src/defaults.rs` (consistent with all other tunable defaults).
- The default of 256 must not break any existing test or real-world use case.

## Acceptance Criteria
- AC1: Both encrypt functions reject > MAX_RECIPIENTS with `GitvaultError::Encryption`.
- AC2: `MAX_RECIPIENTS = 256` defined in `src/defaults.rs`.
- AC3: Validation is deduplicated (single helper or shared guard).
- AC4: Tests for at-limit (accepted) and over-limit (rejected).
- AC5: `recipient add` documents the limit.

## Test Plan
- Unit test: `encrypt_armor` with 256 valid keys → succeeds.
- Unit test: `encrypt_armor` with 257 valid keys → `Err(Encryption(...))`.
- Unit test: `encrypt_binary_b64` with 257 valid keys → same error.
- Unit test: error message contains the actual count and limit value.

## Notes
256 is a conservative default. If a future use case requires more recipients, the constant can be increased or made configurable via `[limits] max_recipients` in `.gitvault/config.toml`. The spec does not require config-file tunability in the initial implementation — the constant is sufficient. The check against zero recipients (`At least one recipient required`) remains as a separate guard below the new limit check.
