## Requirement
REQ-85

## Context
`prompt_prod_confirmation()` in `src/barrier.rs` has a `#[cfg(test)]` branch:

```rust
#[cfg(test)]
{
    let response = std::env::var("GITVAULT_TEST_CONFIRM").unwrap_or_else(|_| "n".to_string());
    let mut input = std::io::Cursor::new(format!("{response}\n").into_bytes());
    read_confirmation_from(&mut input)
}
```

This design has three problems:

1. **`unsafe` env mutation**: Tests that set `GITVAULT_TEST_CONFIRM` use `unsafe { std::env::set_var(...) }`. `set_var` is not thread-safe (Rust 2024 edition marks it `unsafe`). A `Mutex` is used to serialize access, but this only works within the same test binary; it does not protect against cross-binary or cross-process races.

2. **Fragile coupling**: The test behaviour depends on a globally-visible env var. Any test that forgets to clean up the env var (e.g. on panic) will poison subsequent tests.

3. **Unnecessary complexity**: `check_prod_barrier_with_confirm()` already accepts an injectable `confirm: F` closure — the entire `GITVAULT_TEST_CONFIRM` mechanism is a workaround for tests that call the public `check_prod_barrier()` instead of the testable `_with_confirm` variant. The fix is simply to use the already-existing injection point in those tests.

## Goal
Remove the test-only env-var hook from production/test code, eliminate `unsafe` env mutations in tests, and rely exclusively on the existing dependency-injection API (`check_prod_barrier_with_confirm`) for barrier testing.

## Non-goals
- Changing `check_prod_barrier_with_confirm`'s signature or behaviour.
- Adding new test utilities or test helpers.
- Modifying the production `check_prod_barrier` public API.

## Constraints
- All existing test scenarios must be preserved — only the mechanism for injecting the confirmation answer changes (env var → closure).
- The public `check_prod_barrier` API signature is unchanged.
- No `#[cfg(test)]` blocks remain in the implementation of `prompt_prod_confirmation`.

## Acceptance Criteria
- AC1: `GITVAULT_TEST_CONFIRM` env var reading removed from `prompt_prod_confirmation`.
- AC2: Affected tests use `check_prod_barrier_with_confirm` with closures.
- AC3: `GITVAULT_TEST_CONFIRM_LOCK` mutex removed.
- AC4: All barrier tests pass; test count unchanged.
- AC5: No `unsafe { set_var/remove_var }` in barrier test module.

## Test Plan
- `cargo test src/barrier` → all tests pass.
- `grep -n "GITVAULT_TEST_CONFIRM\|set_var\|remove_var\|unsafe" src/barrier.rs` → no matches.
- `cargo clippy -D warnings` → clean.

## Notes
This is a 🟡 LOW severity finding but has immediate value because it removes `unsafe` from the test suite. The implementation is a mechanical refactor: replace `check_prod_barrier(...)` calls in two tests with `check_prod_barrier_with_confirm(..., || Ok(true))` and `check_prod_barrier_with_confirm(..., || Ok(false))` respectively. Estimated effort: 15–20 minutes.
