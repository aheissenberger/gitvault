## Requirement
REQ-73

## Context
`GITVAULT_IDENTITY` currently accepts raw `AGE-SECRET-KEY-…` values inline. This exposes the private key in `/proc/<PID>/environ` and `ps auxe` output for the lifetime of the process — a measurable exposure window. File-descriptor passing avoids this: the FD integer in the environment is not secret material; only the data read through the FD is sensitive, and it is consumed in process memory without appearing in any process-visible env var.

## Goal
Provide a secure, ergonomic alternative to inline env var key passing that is compatible with shell heredocs, process substitution, `systemd` `LoadCredential=`, Docker `--secret`, and similar FD-aware facilities.

## Non-goals
- Windows FD passing (document limitation; recommend temp file on `/dev/shm` equivalent or `GITVAULT_IDENTITY` pointing to a file path instead).
- Replacing or removing `GITVAULT_IDENTITY` inline support (backward compatibility is preserved).
- Interactive TTY key entry via FD.

## Constraints
- FD is consumed once and closed immediately after reading; it must not be held open for the process lifetime.
- Key material must be stored in `Zeroizing<String>` from the moment it is read.
- `unsafe { File::from_raw_fd(n) }` is the correct Unix primitive; it must be documented with a safety comment explaining the caller guarantees (valid open FD, not stdin/stdout/stderr).
- If `GITVAULT_IDENTITY_FD` is set but `GITVAULT_IDENTITY_FD` is empty or non-numeric, treat as configuration error and fail immediately with exit code 2.

## Acceptance Criteria
- AC1: FD-based identity loading reads key correctly and zeroizes on drop.
- AC2: Updated precedence chain is enforced across all identity-consuming command paths.
- AC3: Unreadable FD fails closed; does not silently fall through.
- AC4: `check` command surfaces `GITVAULT_IDENTITY_FD` source state.
- AC5: Windows emits warning and skips; no compile-time Unix APIs exposed.
- AC6: Feature is `#[cfg(unix)]` gated.

## Test Plan
- Unix unit test: write a known key to a `pipe2` or `memfd`; confirm `load_identity_from_fd` returns the correct key material.
- Unix unit test: pass a closed FD; confirm fail-closed error (not fallthrough).
- Unix unit test: pass a non-numeric value; confirm exit code 2 configuration error.
- Integration test (Unix CI): `GITVAULT_IDENTITY_FD=3 gitvault decrypt 3<<<"$KEY"` roundtrip.
- Windows CI: set `GITVAULT_IDENTITY_FD`; confirm warning on stderr and resolution via next source.

## Usage Examples

```bash
# Pipe from a secrets manager (key never in env var)
vault kv get -field=age_key kv/gitvault | GITVAULT_IDENTITY_FD=3 gitvault decrypt 3<&0

# Bash herestring (key in bash variable, not in child env)
GITVAULT_IDENTITY_FD=3 gitvault decrypt 3<<<"$MY_AGE_KEY"

# systemd service with LoadCredential= (FD assigned by systemd)
# LoadCredential=gitvault-key:/run/credentials/gitvault.key
# GITVAULT_IDENTITY_FD=3  (systemd assigns the FD)
```

## Notes
`GITVAULT_IDENTITY_FD` is the recommended approach for non-GitHub-Actions CI and for systemd/Docker deployments. For GitHub Actions, the recommended pattern remains writing to `/dev/shm` (tmpfs) and passing the path via `GITVAULT_IDENTITY`, or using OIDC + an external vault (see `29-identity-security-improvements.md`).
