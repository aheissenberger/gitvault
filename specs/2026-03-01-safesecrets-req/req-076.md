## Requirement
REQ-76

## Context
`GITVAULT_IDENTITY_PASSPHRASE` exposes the raw SSH key passphrase in `/proc/<PID>/environ` and `ps auxe` output for the lifetime of the process. Unlike `GITVAULT_IDENTITY`, there is no file-path alternative for the passphrase — it has always been a raw value. The OS keyring (`gitvault keyring set-passphrase`) is the existing secure alternative for interactive/persistent use, but it requires prior setup and is unavailable in ephemeral CI runners without a keyring daemon. File-descriptor passing fills the CI gap: the passphrase arrives in a pipe/memfd without being placed in any env var visible to the OS process table.

Note: A `--passphrase-stdin` flag is not provided because it would conflict with `--identity-stdin` (both cannot share file descriptor 0 simultaneously). `GITVAULT_IDENTITY_PASSPHRASE_FD` uses a caller-chosen FD number, avoiding this conflict.

## Goal
Provide a secure, non-interactive alternative to `GITVAULT_IDENTITY_PASSPHRASE` for CI and scripted contexts on Unix, without requiring the OS keyring.

## Non-goals
- Windows FD passing (document limitation; recommend OS keyring as the secure alternative on Windows, or use a native age key which requires no passphrase).
- Replacing or removing `GITVAULT_IDENTITY_PASSPHRASE` (backward compatibility preserved).
- A `--passphrase-stdin` flag (conflicts with `--identity-stdin`; FD approach is the correct solution).
- Age native key passphrases (age native keys do not support passphrases; this feature applies only to SSH identity passphrases).

## Constraints
- FD is consumed once and closed immediately after reading; not held open for the process lifetime.
- Passphrase must be stored in `Zeroizing<String>` from the moment it is read.
- `unsafe { File::from_raw_fd(n) }` is the correct Unix primitive; must be documented with a safety comment.
- If `GITVAULT_IDENTITY_PASSPHRASE_FD` is set but empty or non-numeric, treat as configuration error, fail immediately with exit code 2.
- The FD used for the passphrase must not collide with the FD used for `GITVAULT_IDENTITY_FD` — callers are responsible for choosing distinct FD numbers; the tool does not enforce this but should document the requirement.

## Acceptance Criteria
- AC1: FD-based passphrase loading reads correctly and zeroizes on drop.
- AC2: Updated passphrase resolution chain is enforced in `try_fetch_ssh_passphrase` and all callers.
- AC3: Unreadable FD fails closed; does not silently fall through.
- AC4: `check` command surfaces `GITVAULT_IDENTITY_PASSPHRASE_FD` source state.
- AC5: Windows emits warning and skips; no Unix-only API in Windows compile path.
- AC6: Feature is `#[cfg(unix)]` gated.

## Test Plan
- Unix unit test: write a known passphrase to a `pipe` or `memfd`; confirm `load_passphrase_from_fd` returns the correct value.
- Unix unit test: pass a closed FD; confirm fail-closed error (no fallthrough to `GITVAULT_IDENTITY_PASSPHRASE`).
- Unix unit test: pass a non-numeric value; confirm exit code 2 configuration error.
- Integration test (Unix CI): `GITVAULT_IDENTITY_FD=3 GITVAULT_IDENTITY_PASSPHRASE_FD=4 gitvault decrypt 3<<<"$KEY" 4<<<"$PASS"` roundtrip with passphrase-protected SSH identity.
- Windows CI: set `GITVAULT_IDENTITY_PASSPHRASE_FD`; confirm warning on stderr, resolution via `GITVAULT_IDENTITY_PASSPHRASE`.

## Usage Examples

```bash
# Pipe both identity and passphrase via separate FDs (neither in env vars)
GITVAULT_IDENTITY_FD=3 GITVAULT_IDENTITY_PASSPHRASE_FD=4 \
  gitvault decrypt 3<<<"$MY_SSH_KEY" 4<<<"$MY_PASSPHRASE"

# From a secrets manager that can write to named FDs
vault kv get -field=ssh_key kv/gitvault \
  | GITVAULT_IDENTITY_FD=3 GITVAULT_IDENTITY_PASSPHRASE_FD=4 \
    gitvault decrypt 3<&0 4<<<"$(vault kv get -field=passphrase kv/gitvault)"

# systemd service with two LoadCredential= entries
# LoadCredential=gitvault-key:/run/credentials/gitvault.key
# LoadCredential=gitvault-pass:/run/credentials/gitvault.pass
# GITVAULT_IDENTITY_FD=3 GITVAULT_IDENTITY_PASSPHRASE_FD=4
```

## Notes
If a passphrase-protected SSH identity is used in CI and the OS keyring is not available, the recommended fallback order is:
1. `GITVAULT_IDENTITY_PASSPHRASE_FD` (Unix) — passphrase never in env
2. `GITVAULT_IDENTITY_PASSPHRASE` (all platforms) — passphrase in env (with deprecation warning per REQ-77)
3. Switch to a native age identity (no passphrase required) — eliminates the problem entirely
