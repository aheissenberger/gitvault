## Requirement
REQ-84

## Context
`encrypt_armor` and `encrypt_binary_b64` in `src/structured/armor.rs` format invalid-key errors as:

```rust
format!("Invalid recipient key {k}: {e}")
```

where `k` is the full `age1...` public key string (59 characters). While age recipient public keys are not private secrets (they are designed to be shared), including the full key in error messages has two problems:

1. **Log pollution**: If error messages are logged centrally, full public keys appear in log stores — increasing the attack surface for social engineering or key enumeration (an attacker who collects public keys can target their encrypted outputs).
2. **Consistency with spec 11 note**: The existing spec states "Logging must be audited to avoid accidental value disclosure." Truncating key material in errors is part of this audit.

The first 8 characters of an age public key (`age1xxxx`) are sufficient to identify a key in a team context without exposing the full key.

## Goal
Reduce key material in error messages to a non-identifying truncated fingerprint, consistent with the principle of minimum disclosure in log output.

## Non-goals
- Hashing or cryptographically deriving the fingerprint (simple prefix truncation is sufficient for this low-severity finding).
- Removing error messages entirely (they must remain actionable).
- Redacting key material from `--json` output fields that are specifically documented to contain key data (e.g., `gitvault identity pubkey` output).

## Constraints
- The truncation helper must be a named function (not inline string slice) for testability.
- The error must remain actionable: the truncated prefix is sufficient to let a user identify which key in their config is invalid.
- This is a 2–3 line change plus a helper function. No refactoring of error types.

## Acceptance Criteria
- AC1: `{k}` replaced with `truncate_key_for_log(k)` in both error format strings.
- AC2: `truncate_key_for_log` is a testable standalone function.
- AC3: No new key material in other error messages.
- AC4: Tests updated and new tests for the helper added.

## Test Plan
- Unit test: `truncate_key_for_log("age1qfr8ydqtf7j2e8q...")` → `"age1qfr8…"`.
- Unit test: `truncate_key_for_log("age1abc")` → `"age1abc"` (shorter than 8 chars, no truncation).
- Updated test: `test_encrypt_armor_invalid_key_errors` → error contains `"Invalid recipient key"` and `"…"` but not the full input key.
