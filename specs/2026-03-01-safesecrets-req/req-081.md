## Requirement
REQ-81

## Context
`check_no_tracked_plaintext` in `src/repo/drift.rs` (REQ-10) only checks **staged** files (`git diff --cached --name-only`). It does not scan committed history. A user who accidentally committed a `.env` file or a file under `.secrets/plain/` and later removed it or encrypted it has residual plaintext in git history — readable by anyone who clones the repo.

This is a meaningful security gap: the pre-commit hook catches new staging events, but does not alert users to already-committed plaintext that persists in history. The `gitvault check` preflight command is the correct place to surface this because it is explicitly designed to report the full security state of the repository.

## Goal
Surface committed-history plaintext leaks in `gitvault check`, so users are alerted to the need for `git filter-repo` (or equivalent) history rewriting, before they share the repository.

## Non-goals
- Automatically rewriting history (out of scope; risky and destructive).
- Scanning file contents in history (only path-based detection, same as staged check).
- Replacing `check_no_tracked_plaintext` (staged check remains).
- Checking binary blobs or encrypted `.age` files.

## Constraints
- `git log` invocation must be bounded (HISTORY_SCAN_LIMIT) to remain performant.
- The check must not fail loudly in repos with no commits (fresh init).
- Output format must be consistent with existing `check` command output.
- `--skip-history-check` must be documented in `gitvault check --help`.

## Acceptance Criteria
- AC1: `check_history_for_plaintext` correctly identifies files matching plaintext patterns in history.
- AC2: `gitvault check` surfaces history leaks; exits non-zero when found.
- AC3: Staged check (`check_no_tracked_plaintext`) unchanged.
- AC4: Scan bounded at HISTORY_SCAN_LIMIT commits; truncation warning emitted.
- AC5: No-commit repos and missing-git cases handled gracefully.
- AC6: Skip flag/env var works; notice emitted when skipped.

## Test Plan
- Unit test: repo with a `.env` file committed and then deleted → `check_history_for_plaintext` returns it.
- Unit test: repo with only encrypted `.age` files → returns empty list.
- Unit test: empty repo (no commits) → returns empty list (no error).
- Unit test: `--skip-history-check` → function not called; notice emitted.
- Integration test: `gitvault check` exit code 1 when history leak present; `--json` output contains `history_leaks` array.

## Notes
The recommended remediation for committed plaintext is `git filter-repo --path <path> --invert-paths` followed by force-pushing all branches. This should be documented in the warning message. `BFG Repo Cleaner` is an alternative. The check output should include the specific git command needed, not just a generic "rewrite history" suggestion.
