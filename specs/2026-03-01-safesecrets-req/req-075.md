## Requirement
REQ-75

## Context
Inline `AGE-SECRET-KEY-…` values in `GITVAULT_IDENTITY` are exposed in `/proc/<PID>/environ` (Linux), `ps auxe` output, and any process-environment dump tool for the lifetime of the process. While the tool continues to support this for backward compatibility, users who are unaware of the risk may use it in production systems (including CI runners). A deprecation warning at runtime, directed to stderr, gives users actionable guidance without breaking their workflows.

## Goal
Make users aware of the security risk of inline key material in environment variables, and point them to the two safer alternatives (`GITVAULT_IDENTITY_FD` and `--identity-stdin`) without removing backward compatibility.

## Non-goals
- Removing inline key support.
- Changing exit codes or blocking execution.
- Warning about `GITVAULT_IDENTITY` when it contains a file path (file paths are the safe pattern).
- Warning about `GITVAULT_IDENTITY_PASSPHRASE` (separate issue; not in scope here).

## Constraints
- Warning must go to stderr only; never stdout or structured JSON output.
- Warning detection is purely syntactic: value starts with `AGE-SECRET-KEY-` (case-insensitive for robustness, though the `age` crate always produces uppercase).
- The `GITVAULT_NO_INLINE_KEY_WARN=1` suppression escape hatch is exclusively for test infrastructure; it must not be documented in user-facing help or README as a production option.

## Acceptance Criteria
- AC1: Warning is emitted to stderr when inline key is detected.
- AC2: Warning is suppressed in `--json` mode; not emitted for file paths.
- AC3: Inline key still resolves correctly (no regression).
- AC4: Warning text is stable and matches the documented format.
- AC5: Test suite does not generate spurious warnings; suppression escape hatch works.

## Test Plan
- Unit test: `GITVAULT_IDENTITY=AGE-SECRET-KEY-…` → stderr contains `[gitvault WARN]` line; exit 0.
- Unit test: `GITVAULT_IDENTITY=/path/to/key.age` → no warning on stderr.
- Unit test: `GITVAULT_IDENTITY=AGE-SECRET-KEY-… --json` → no warning on stderr.
- Unit test: `GITVAULT_NO_INLINE_KEY_WARN=1 GITVAULT_IDENTITY=AGE-SECRET-KEY-…` → no warning.
- Regression: existing inline-key tests pass without stderr noise (use suppression env var in test setup).

## Notes
This warning is the lowest-effort security improvement in the `GITVAULT_IDENTITY` hardening set. It should be implemented first (before or alongside REQ-73/REQ-74) because it delivers immediate risk visibility to existing users with zero breaking changes. The full priority order for the GITVAULT_IDENTITY security improvement set is: REQ-75 (warn) → REQ-74 (stdin) → REQ-73 (FD).
