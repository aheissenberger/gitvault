## Overview
The HMAC signature over a token's expiry timestamp must use a canonical, deterministic encoding to avoid side-channel attacks and ensure consistent verification. The current implementation uses `expiry.to_string().as_bytes()`, which is locale-dependent and non-canonical. This spec changes to `expiry.to_be_bytes()` (8 big-endian bytes), a binary, fixed-length encoding.

## Key Changes

### compute_hmac Function
```rust
// Before
mac.update(expiry.to_string().as_bytes());

// After
mac.update(&expiry.to_be_bytes());
```

The 8-byte big-endian representation is deterministic and immune to locale or formatting variations.

## Breaking Change
All existing allow-prod tokens become invalid because their HMAC was computed with the old encoding. When a token fails verification, `allow_prod` prompts the user to re-issue a new token with `gitvault allow-prod`, which computes the HMAC with the new encoding. This is the intended behavior: the barrier fails closed on mismatch.

## Testing
- **New test cases:** Verify that tokens issued with the new encoding pass verification.
- **Legacy token tests:** Verify that tokens issued with the old encoding fail verification.
- **Round-trip test:** Issue a token, verify it immediately, and confirm the HMAC matches.

## Risk & Mitigation
- **One-time user friction:** Users with existing tokens must re-issue them once.
- **Security gain:** Canonical encoding prevents locale-based attacks and ensures cross-platform consistency.
