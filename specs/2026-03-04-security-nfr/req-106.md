## Problem

Secrets are stored with arbitrary key names. When `gitvault materialize` writes them to `.env`,
the `escape_env_value` function correctly escapes the *value* side, but the *key* side is
written verbatim from the stored key name.

A key name containing shell metacharacters could produce malformed `.env` output that is
misinterpreted by shell `source` commands or poorly-written `.env` parsers.

Example: a stored key `FOO\nBAR` would produce:
```
FOO
BAR="value"
```
in the `.env` file, potentially clobbering the output structure.

## Solution

Validate key names against the POSIX shell variable name pattern `[A-Za-z_][A-Za-z0-9_]*`
at the point where a new key is accepted (encrypt, set, add). Existing stored keys that were
accepted before this validation are not re-validated at materialize time (to avoid breaking
existing repos).

## Implementation Status

**Status: specified** — validation function to be implemented in a future patch.

The planned implementation location is `src/merge.rs` (alongside the existing `parse_env_pairs`
and `parse_env_key_from_line` functions), with call sites in `src/structured/fields.rs` and
`src/commands/encrypt.rs`.
