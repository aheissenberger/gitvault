# REQ-113: Atomic Write Consolidation (extends REQ-107)

**ID:** S-20260304-R0113
**Status:** Implemented
**Priority:** High
**Category:** Security / Reliability

## Problem

REQ-107 introduced `fs_util::atomic_write` as the canonical atomic file-write utility and
migrated the initial set of call sites. Three production write sites were not included in that
initial migration:

| File | Write target | Prior implementation |
|------|-------------|---------------------|
| `src/repo/hooks.rs` | git hook scripts | `std::fs::write` (non-atomic) |
| `src/commands/materialize.rs` | `.gitignore`, `.gitattributes` | `std::fs::write` or inline `NamedTempFile` |
| `src/recipients.rs` | recipients list | `std::fs::write` or inline `NamedTempFile` |

Each site has a distinct failure mode if the write is interrupted mid-operation (process
kill, power loss, or disk full):

- **Hooks** — truncated hook body → silent bypass of the pre-commit secret-leak guard.
- **`.gitignore`** — truncated ignore rules → transient exposure of ignored paths to `git add`.
- **Recipients** — truncated list → ciphertext unreadable by some recipients, or incorrect
  recipient set used for the next encryption.

Additionally, any inline `NamedTempFile`-plus-rename logic outside `fs_util.rs` duplicates
the utility's implementation, risking divergence (e.g., temp file placed in `/tmp` instead of
the target directory, causing a cross-device rename that falls back to a non-atomic copy).

## Requirement

All three sites MUST be migrated to `fs_util::atomic_write`. No inline `NamedTempFile` or
`std::fs::write`-without-rename pattern is permitted for security-sensitive file targets in
production code. Sites that cannot use `fs_util::atomic_write` (e.g., because a permission
step is required between write and rename) MUST carry a documented exception comment.

## Implementation

Each of the three sites is updated to replace the previous write pattern with:

```rust
fs_util::atomic_write(&target_path, content.as_bytes())?;
```

Where `content` is the complete file content (hook body, `.gitignore` text, recipients list).

The `fs_util::atomic_write` function:
1. Creates a `NamedTempFile` in the **same directory** as the target (same filesystem mount),
   ensuring the final `persist()` rename is always atomic on POSIX systems.
2. Writes all bytes to the temp file.
3. Renames (persists) the temp file to the target path.
4. Returns `GitvaultError` on any failure, without leaving partial content at the target path.

## Acceptance Criteria Summary

- **AC1** — `hooks.rs` hook writes use `fs_util::atomic_write`.
- **AC2** — `materialize.rs` `.gitignore` / `.gitattributes` writes use `fs_util::atomic_write`.
- **AC3** — `recipients.rs` recipients file write uses `fs_util::atomic_write`.
- **AC4** — No duplicate `NamedTempFile`-plus-rename logic in production code outside `fs_util.rs`.
- **AC5** — Existing tests for all three modules pass unchanged after migration.
