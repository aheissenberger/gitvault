## Problem

### SEC-003 — Env variable name size limit

`GITVAULT_ENV` and any equivalent env-file override supply the environment name used to
select the active secret environment. The name is used as a path component in the repository
layout (`.git/gitvault/<env>/`). If the name is unboundedly long it can:

1. Exceed filesystem path limits and cause confusing I/O errors far from the input boundary.
2. Differ in behaviour between the env-var path and the env-file path if only one path
   enforces a length check, creating inconsistent behaviour.

### SEC-007 — Secret key name characters

Secret keys are stored in encrypted form and later written to `.env` output by
`gitvault materialize`. The value side is correctly escaped (REQ-106), but the *key* side is
taken verbatim from the stored name. Key names containing:

- `=` — produce `KEY=REST=value` which is ambiguous: some parsers use the first `=`, others
  the last.
- NUL (`\0`) — truncate the key name at the C-string boundary in POSIX-compliant parsers,
  potentially colliding with a shorter, legitimately-named key.
- Empty string — produce `=value`, which most parsers silently skip or error on.

## Solution

### Part A — Env name size cap (SEC-003)

At each site where the env name is read from the environment or an env file, apply a 255-byte
length check. Names that exceed the limit are treated as absent and the compiled-in default
(`"default"`) is used instead. No error is surfaced to the user: the fallback is safe and an
excessively long name is almost certainly a misconfiguration rather than a deliberate choice.

255 bytes is chosen because:
- It equals the maximum filename component length on most filesystems (ext4, APFS, NTFS).
- It is the same limit applied to the dual use of the env name as a directory component.

### Part B — Secret key name validation (SEC-007)

At every write boundary (file encryption, structured field set/add), validate the key name
before persisting:

```rust
fn validate_secret_key_name(key: &str) -> Result<(), GitvaultError> {
    if key.is_empty() {
        return Err(GitvaultError::Usage(
            "secret key name must not be empty".into(),
        ));
    }
    if key.contains('=') {
        return Err(GitvaultError::Usage(format!(
            "secret key name '{key}' must not contain '='"
        )));
    }
    if key.contains('\0') {
        return Err(GitvaultError::Usage(format!(
            "secret key name '{key}' must not contain a NUL byte"
        )));
    }
    Ok(())
}
```

## Implementation

- **`src/config.rs`** — env name length check on the `GITVAULT_ENV` env-var read path.
- **`src/config.rs`** (env-file parse path) — same length check applied after reading the
  env name from the env file, ensuring both paths are consistent.
- **`src/merge.rs`** — `validate_secret_key_name` function added.
- **`src/structured/fields.rs`** — calls `validate_secret_key_name` before writing a
  structured field.
- **`src/commands/encrypt.rs`** — calls `validate_secret_key_name` for each key in the
  input file before any encryption is performed.

## Acceptance Criteria Summary

- **AC1** — Names > 255 bytes fall through to the compiled-in default; no panic or I/O error.
- **AC2** — The 255-byte limit is applied on both the env-var and env-file code paths.
- **AC3** — Empty key, key with `=`, and key with NUL all return `GitvaultError::Usage`.
- **AC4** — Validation occurs before any file write (fail-closed).
- **AC5** — Tests cover boundary cases for both the env name limit and the key name rules.
