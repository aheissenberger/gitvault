## Overview
The `find_history_plaintext_leaks` function in `src/repo/drift.rs` scans the repository's git history to detect plaintext secrets (e.g., passwords in `.env` files). Currently it uses `--diff-filter=A` (Added) to find only files that were newly created. However, an attacker could rename a non-sensitive file (e.g., `config.txt`) to a sensitive one (e.g., `.env`) and then commit plaintext secrets in the renamed file. The Renamed filter would not detect the rename itself as adding a sensitive file. This spec adds the Renamed filter (`R`) to catch files that are renamed into sensitive paths.

## Key Changes

### Before
```bash
git log --diff-filter=A --name-only -- ':!target/**'
```

### After
```bash
git log --diff-filter=AR --name-only -- ':!target/**'
```

The `AR` filter includes both Added and Renamed files, catching:
1. Files explicitly added with a sensitive name (e.g., `git add .env`).
2. Files renamed to a sensitive name (e.g., `git mv config.txt .env`).

## Detection Logic
When a file is renamed, `git log --diff-filter=AR` shows both the old and new names. The scan checks the new name against the sensitive path patterns (e.g., `.env`, `.git/config`). If the new name matches, the rename is flagged as a potential leak.

## Testing

### New Test: Rename to Sensitive Name
```
1. Create a file: `git add config.txt`
2. Commit it: `git commit -m "add config"`
3. Rename it: `git mv config.txt .env`
4. Commit the rename: `git commit -m "rename to .env"`
5. Run the history scan: `find_history_plaintext_leaks()`
6. Verify that the rename is detected as a potential leak.
```

### Existing Tests
All existing tests that check for Added files continue to pass unchanged.

## Behavior Change
- **Scope expansion:** The scan now catches renames to sensitive paths, in addition to direct additions.
- **False positives:** Low risk; files are only flagged if renamed **to** a sensitive name.
- **False negatives:** Renames from a sensitive name to a non-sensitive name are **not** flagged (expected; the data is no longer in a sensitive location).

## Risk & Mitigation
- **No breaking changes:** The function's return type and interface remain identical.
- **More comprehensive:** Improved detection of both obfuscated and renamed leaks.
