## Overview
The `allow_prod` command issues time-limited access tokens for interacting with protected repositories. Without a hard limit on token lifetime, a user could request a token valid for days or weeks, increasing the window of vulnerability if the token is compromised. This spec defines a maximum token lifetime of 24 hours and enforces it as a hard limit that cannot be overridden.

## Key Changes

### Define the Constant
```rust
pub const MAX_TOKEN_TTL_SECS: u64 = 86_400;  // 24 hours in seconds
```

Place this constant at the top of `src/barrier.rs` for visibility and easy adjustment if needed in the future.

### Enforce in allow_prod()
```rust
pub fn allow_prod(ttl_secs: u64) -> Result<String, GitvaultError> {
    // Validate TTL before any file I/O
    if ttl_secs > MAX_TOKEN_TTL_SECS {
        return Err(GitvaultError::Usage(
            format!("TTL {}s exceeds maximum {}s (24 h)", ttl_secs, MAX_TOKEN_TTL_SECS)
        ));
    }
    
    // ... load or create key, compute HMAC, etc. ...
}
```

The check is performed **before** any file operations (key loading, token writing) to fail fast and avoid unnecessary I/O.

## Error Handling
When the TTL exceeds the limit, the error is returned as `GitvaultError::Usage`, indicating a user error. The error message is clear and actionable:
```
TTL 172800s exceeds maximum 86400s (24 h)
```

This helps users understand why their request failed and what the constraint is.

## Testing

### Unit Test: TTL Exceeds Maximum
```rust
#[test]
fn test_allow_prod_ttl_exceeds_max() {
    let result = allow_prod(MAX_TOKEN_TTL_SECS + 1);
    assert!(result.is_err());
    let err_msg = result.unwrap_err().to_string();
    assert!(err_msg.contains("exceeds maximum"));
    assert!(err_msg.contains("86400"));
}
```

### Unit Test: TTL Within Limit
```rust
#[test]
fn test_allow_prod_ttl_within_limit() {
    let result = allow_prod(3600);  // 1 hour
    assert!(result.is_ok());
}
```

### Edge Case: TTL Equals Maximum
```rust
#[test]
fn test_allow_prod_ttl_equals_max() {
    let result = allow_prod(MAX_TOKEN_TTL_SECS);
    assert!(result.is_ok());
}
```

## Default TTL
The default TTL (typically 3600 seconds = 1 hour) is well within the 24-hour limit. Existing callers and workflows are unaffected.

## Security Rationale
A 24-hour limit strikes a balance between usability and security:
- **Short enough** to minimize risk if a token is compromised.
- **Long enough** for typical CI/CD workflows and manual interventions.
- **Well-understood:** 24 hours is a common TTL in OAuth 2.0 and other token systems.

## Risk & Mitigation
- **No breaking changes:** Existing tokens with short TTLs (< 24 h) continue to work.
- **Clear error message:** Users who request longer TTLs receive actionable feedback.
- **Policy enforcement:** The cap is hardcoded and cannot be bypassed via environment variables or config files.
