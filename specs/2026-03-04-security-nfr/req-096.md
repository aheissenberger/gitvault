## Overview
The `cmd_recipient_add_self` command in `src/commands/recipients.rs` may call `git config user.name` multiple times: once in a helper function to derive the recipient name, and again during idempotency checking. Each call spawns a subprocess, which is expensive. This spec optimizes the command to fetch the user name once and reuse it for all downstream operations.

## Key Changes

### cmd_recipient_add_self Refactoring
```rust
pub fn cmd_recipient_add_self(/* ... */) -> Result<(), GitvaultError> {
    // Fetch the user name once
    let user_name = get_git_config_user_name()?;  // Single subprocess call
    
    // Reuse for idempotency check
    if self_recipient_exists(&user_name)? {
        println!("Recipient already registered: {}", user_name);
        return Ok(());
    }
    
    // Reuse for filename derivation
    let derived_name = sanitize_name(&user_name);
    let pub_file = format!("{}.pub", derived_name);
    
    // ... rest of function ...
}
```

### Refactor derive_recipient_name()
The standalone `derive_recipient_name()` function that spawns `git config user.name` is replaced with a parameter-based version:

**Before:**
```rust
fn derive_recipient_name() -> Result<String, GitvaultError> {
    let user_name = run_git_config("user.name")?;  // Subprocess
    Ok(sanitize_name(&user_name))
}
```

**After:**
```rust
fn derive_recipient_name(user_name: &str) -> String {
    sanitize_name(user_name)
}
```

Callers that already have the user name pass it in; callers that don't fetch it once and reuse it.

## Behavior
- **Single subprocess call:** `git config user.name` is invoked exactly once per `cmd_recipient_add_self` invocation.
- **Result reuse:** The sanitized name is used for both idempotency checking and filename derivation.
- **No external behavior change:** The command output and file creation are identical.

## Testing
- **Unit tests:** Verify that the command still creates the `.pub` file with the correct name.
- **Mock testing:** Mock `get_git_config_user_name()` to return a fixed value and verify it is called exactly once.
- **Integration tests:** Confirm that existing recipients are still detected correctly and the command is idempotent.

## Performance
- **Subprocess reduction:** One fewer subprocess spawn per command, improving responsiveness.
- **Negligible overhead:** Each subprocess call takes ~10-50ms on modern systems; saving one call per command is a measurable but small improvement.

## Risk & Mitigation
- **No breaking changes:** Command interface and behavior remain identical.
- **Maintainability:** Clearer code path; single point of fetching the user name.
