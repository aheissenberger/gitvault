## Problem

`run_command` in `src/run.rs` injects all decrypted secrets into the child process environment
unconditionally. If a secret key is named `PATH`, the injected value replaces the system `PATH`
in the child's environment. The child process then looks up binaries using the injected path,
which may point to attacker-controlled directories if the secret was maliciously crafted.

Similar risks apply to `LD_PRELOAD` (injects a shared library into every spawned process) and
`LD_LIBRARY_PATH` (redirects dynamic linker search).

These variable names in a secrets store are almost always an accident: the operator intended to
store a filesystem path as a secret, not override the system `PATH`.

## Solution

Emit a `gitvault: warning:` message to stderr when any injected key matches `PATH`,
`LD_PRELOAD`, or `LD_LIBRARY_PATH`. The injection still proceeds ŌĆö the operator may have an
intentional use case (e.g. `--clear-env` mode where the child needs a minimal `PATH`).

The warning surfaces the risk without breaking any legitimate workflow.

## Implementation

```rust
for (key, _) in secrets {
    if key == "PATH" || key == "LD_PRELOAD" || key == "LD_LIBRARY_PATH" {
        eprintln!(
            "gitvault: warning: secret key '{key}' overrides a sensitive environment \
             variable ŌĆö verify this is intentional"
        );
    }
}
```

Added before `Command::new(cmd)` in `run_command`.
