## Problem

`compute_hmac` in `src/barrier.rs` previously called:

```rust
HmacSha256::new_from_slice(key).expect("HMAC key is always 32 bytes — infallible")
```

While the comment is correct (HMAC-SHA256 accepts a 32-byte key), the `expect()` call:

1. **Panics in production** if the invariant is ever violated (e.g. by refactoring that changes
   the key type from `[u8; 32]` to `Vec<u8>`).
2. **Sets a bad precedent** — other engineers may copy the pattern and apply it to less-safe paths.
3. **Violates NFR-001 AC7** which prohibits `expect()` in non-test production code.

## Solution

Change `compute_hmac` to return `Result<String, GitvaultError>` and use `.map_err(...)` on
`new_from_slice`. Update all callers to propagate the error.

Since the argument type is still `&[u8; 32]`, the error branch is statically unreachable at
runtime — but propagating it is zero-cost and makes the code audit-friendly.

## Implementation

```rust
fn compute_hmac(key: &[u8; 32], expiry: u64) -> Result<String, GitvaultError> {
    let mut mac = HmacSha256::new_from_slice(key)
        .map_err(|_| GitvaultError::Other("HMAC key initialisation failed".into()))?;
    mac.update(&expiry.to_be_bytes());
    let result = mac.finalize().into_bytes();
    Ok(hex::encode(result))
}
```

Callers updated:
- `allow_prod`: `let mac = compute_hmac(&key, expiry)?;`
- `has_valid_token`: `let Ok(expected_mac) = compute_hmac(&key, expiry) else { return false };`
- Tests: `compute_hmac(&key, 1).unwrap()` (acceptable in test code).
