## Problem

`load_or_create_token_key` in `src/barrier.rs` reads the 32-byte HMAC signing key from
`.git/gitvault/.token-key`. On creation (REQ-89) the file is written with mode 0600.
However, this mode can be loosened by:

- The user running `chmod` manually.
- A `umask` misconfiguration that inherits group bits.
- A filesystem backup tool that restores incorrect permissions.
- Git operations that touch `.git/` with different ACLs.

Once the key is world-readable, any local user can forge a production allow-token with an
arbitrary future expiry, bypassing the production barrier permanently without interactive
confirmation.

## Solution

After successfully reading the key file, verify that its mode is restricted to the owner
(`mode & 0o077 == 0`). If not, reject the key with an error that tells the operator to
regenerate it by running `gitvault allow-prod`.

This is a defence-in-depth check: the key was written with 0600 at creation time
(existing `permissions::enforce_owner_rw` call), but this load-time check catches post-write
permission relaxation.

## Implementation

Added to `load_or_create_token_key` in `src/barrier.rs`, in the `Ok(raw)` branch before
the key bytes are decoded:

```rust
#[cfg(unix)]
{
    use std::os::unix::fs::PermissionsExt;
    let mode = fs::metadata(&key_path)?.permissions().mode() & 0o777;
    if mode & 0o077 != 0 {
        return Err(GitvaultError::Other(format!(
            "token key file has insecure permissions (mode {mode:04o}); \
             expected 0600 — run `gitvault allow-prod` to regenerate"
        )));
    }
}
```
