## Problem

`list_ssh_agent_keys` in `src/identity.rs` (or `src/ssh.rs`) parses the output of
`ssh-add -l` and extracts the comment field for each loaded key. This comment is then used as
a filename component when building a path under the gitvault SSH key directory:

```rust
let key_path = ssh_dir.join(&comment);
```

SSH agent key comments are **user-controlled** and are not validated by the SSH agent or by
`ssh-add`. Any process that can access the agent socket can load a key with a comment
containing:

- Forward slashes (`/`) — path traversal on Unix/macOS/Linux.
- Backslashes (`\`) — path traversal on Windows (and confusing on Unix).
- NUL bytes (`\0`) — path aliasing in C-string–based syscall wrappers.
- Leading dots — hidden-file creation, bypassing audit visibility.

Without sanitization, the comment is used directly in `Path::join`, making this a
**path traversal vulnerability** (CWE-22) that could allow a malicious SSH key comment to
cause gitvault to read or write files outside the intended directory.

## Solution

Introduce a `sanitize_comment_for_filename` function that:

1. **Rejects** comments containing `/`, `\`, or `\0` — returns `None` to signal that the
   comment is unsafe as a filename component.
2. **Strips** any number of leading `.` characters to prevent hidden-file creation.
3. **Returns** `Some(sanitized_comment)` for safe comments.

Every path construction that uses an SSH agent key comment calls this function first. If the
function returns `None`, the path is not constructed and the key is skipped (or an error is
returned, depending on context).

## Implementation

```rust
/// Sanitizes an SSH agent key comment for use as a filesystem path component.
///
/// Returns `None` if the comment contains path-separator characters (`/`, `\`) or a NUL
/// byte, indicating the comment cannot safely be used as a filename.
///
/// Leading `.` characters are stripped to prevent hidden-file creation.
pub(crate) fn sanitize_comment_for_filename(comment: &str) -> Option<String> {
    if comment.contains('/') || comment.contains('\\') || comment.contains('\0') {
        return None;
    }
    let stripped = comment.trim_start_matches('.');
    if stripped.is_empty() {
        return None;
    }
    Some(stripped.to_owned())
}
```

Call site (before):
```rust
let key_path = ssh_dir.join(&comment);
```

Call site (after):
```rust
let safe_name = sanitize_comment_for_filename(&comment)?;
let key_path = ssh_dir.join(&safe_name);
```

The `?` operator short-circuits with `None` / an error value appropriate to the calling
context.

## Acceptance Criteria Summary

- **AC1** — `sanitize_comment_for_filename` returns `None` for `/`, `\`, or NUL in comment.
- **AC2** — Leading dots are stripped from the returned name.
- **AC3** — All call sites use `sanitize_comment_for_filename` before `ssh_dir.join()`.
- **AC4** — Tests cover path-traversal attempts, NUL, leading-dot stripping, and benign comments.
- **AC5** — No bare `ssh_dir.join(comment)` with unsanitized comment in production code.
