## Problem

REQ-90 (`src/git.rs`) strips `GIT_DIR`, `GIT_CONFIG`, and `GIT_CONFIG_GLOBAL` before every
git subprocess invocation. However, git respects a much wider set of environment variables
that can redirect its SSH transport, binary lookup, index, and object storage. If any of
these variables is present in the parent process environment (from a CI system, a compromised
`.env` file, or a user mistake), git operations inside gitvault will silently operate against
attacker-controlled paths or binaries.

Additionally, the three previously-stripped variables were repeated verbatim in both the sync
and async helper functions, creating a DRY violation: a future addition to the list must be
made in two places, with no compiler enforcement that the two lists stay in sync.

## Solution

Define a single `GIT_SANITIZE_VARS` constant at module scope in `src/git.rs` containing all
ten variable names. Both `git_run`/`git_output` (sync) and `git_output_async` (async) iterate
over this constant. Adding a new variable to the sanitization list requires a single-line
change at one location.

## Implementation

```rust
/// Environment variables stripped from every git subprocess to prevent environment-based
/// git configuration injection. Both sync and async git helpers iterate this constant.
pub(crate) const GIT_SANITIZE_VARS: &[&str] = &[
    // From REQ-90 (SEC-000)
    "GIT_DIR",
    "GIT_CONFIG",
    "GIT_CONFIG_GLOBAL",
    // SEC-001 additions
    "GIT_SSH_COMMAND",
    "GIT_SSH",
    "GIT_EXEC_PATH",
    "GIT_INDEX_FILE",
    "GIT_OBJECT_DIRECTORY",
    "GIT_ALTERNATE_OBJECT_DIRECTORIES",
    "GIT_WORK_TREE",
];
```

Both sync and async helpers:

```rust
let mut cmd = Command::new("git");
for var in GIT_SANITIZE_VARS {
    cmd.env_remove(var);
}
```
